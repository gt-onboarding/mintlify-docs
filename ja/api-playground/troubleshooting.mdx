API ページが正しく表示されない場合は、以下のよくある設定ミスを確認してください。

<AccordionGroup>
  <Accordion title="OpenAPI ページがすべて空白になります">
    このケースでは、Mintlify が OpenAPI ドキュメントを見つけられないか、
    OpenAPI ドキュメントが無効である可能性が高いです。

    ローカルで `mint dev` を実行すると、これらの問題の一部が判明します。

    OpenAPI ドキュメントが検証を通過することを確認するには:

    1. [このバリデーター](https://editor.swagger.io/)にアクセス
    2. 「Validate text」タブに切り替え
    3. OpenAPI ドキュメントを貼り付け
    4. 「Validate it!」をクリック

    下に表示されるテキストボックスの枠が緑色であれば、ドキュメントは検証に合格しています。
    これは Mintlify が OpenAPI ドキュメントの検証に使用しているものと同じ検証パッケージです。ここで
    検証に合格すれば、問題は別の箇所にある可能性が高いです。

    また、Mintlify は OpenAPI 2.0 をサポートしていません。ドキュメントがこのバージョンの仕様を使用している場合は、
    この問題が発生することがあります。ドキュメントは [editor.swagger.io](https://editor.swagger.io/)（Edit &gt; Convert to OpenAPI 3）で変換できます:

    <Frame>
      ![Swagger Editor のスクリーンショット。Edit メニューが展開され、「Convert to OpenAPI 3」メニュー項目がハイライトされています。](/images/convert-oas-3.png)
    </Frame>
  </Accordion>

  <Accordion title="OpenAPI ページのうち一部がまったく空白になります">
    これは通常、ページのメタデータ内の `openapi` フィールドの綴り誤りが原因です。HTTP メソッドとパスが、OpenAPI ドキュメント内の HTTP メソッドおよびパスと完全に一致していることを確認してください。

    問題が起こりうる例を次に示します。

    ```mdx get-user.mdx
    ---
    openapi: "GET /users/{id}/"
    ---
    ```

    ```yaml openapi.yaml
    paths:
      "/users/{id}":
        get: ...
    ```

    `openapi` フィールドのパスには末尾にスラッシュがありますが、OpenAPI ドキュメント側のパスにはありません。

    もう一つのよくある問題は、ファイル名の綴り誤りです。`openapi` フィールドで特定の OpenAPI ドキュメントを指定している場合は、ファイル名が正しいことを確認してください。たとえば、OpenAPI ドキュメントが `openapi/v1.json` と `openapi/v2.json` の2つある場合、メタデータは次のようになります：

    ```mdx api-reference/v1/users/get-user.mdx
    ---
    openapi: "v1 GET /users/{id}"
    ---
    ```
  </Accordion>

  <Accordion title="APIプレイグラウンドからのリクエストが機能しない">
    カスタムドメインを設定している場合、リバースプロキシが原因の可能性があります。デフォルトでは、APIプレイグラウンド経由のリクエストは、ドキュメントサイト上の `/_mintlify/api/request` パスへの `POST` リクエストから始まります。リバースプロキシが `GET` リクエストのみ許可する設定になっていると、これらのリクエストはすべて失敗します。解決するには、`/_mintlify/api/request` パスへの `POST` リクエストを許可するようリバースプロキシを設定してください。

    また、リバースプロキシの制約で `POST` リクエストを受け付けられない場合は、[設定ドキュメント](/ja/organize/settings#param-proxy)にあるとおり、`docs.json` の `api.playground.proxy` 設定を用いて、Mintlify がプロキシを介さずバックエンドに直接リクエストを送るよう構成できます。この構成を使用する場合、リクエストはプロキシ経由ではなくユーザーのブラウザから直接送信されるため、サーバー側で CORS を設定する必要があります。
  </Accordion>

  <Accordion title="OpenAPI のナビゲーション項目でページが生成されない">
    OpenAPI のナビゲーション構成を使用しているのにページが生成されない場合、次のよくある原因を確認してください。

    1. **デフォルトの OpenAPI 仕様がない**: ナビゲーション要素に `openapi` フィールドが設定されていることを確認してください。

    ```json {5}
    "navigation": {
      "groups": [
        {
          "group": "APIリファレンス",
          "openapi": "/path/to/openapi.json",
          "pages": [
            "GET /users",
            "POST /users"
          ]
        }
      ]
    }
    ```

    2. **OpenAPI 仕様の継承**: ネストされたナビゲーションを使用している場合、子グループが正しい OpenAPI 仕様を継承しているか、あるいは独自の仕様を指定していることを確認してください。

    3. **検証に関する問題**: `mint openapi-check <path-to-openapi-file>` を使用して、OpenAPI ドキュメントが有効かどうかを検証してください。
  </Accordion>

  <Accordion title="一部の OpenAPI のオペレーションはナビゲーションに表示されますが、表示されないものもあります">
    1. **非表示のオペレーション**: OpenAPI スペックで `x-hidden: true` が付与されたオペレーションは、自動生成されたナビゲーションに表示されません。
    2. **無効なオペレーション**: OpenAPI スペックで検証エラーのあるオペレーションはスキップされる場合があります。OpenAPI ドキュメントに構文エラーがないか確認してください。
    3. **手動と自動の取り込み**: OpenAPI スペックのエンドポイントを参照する場合、ナビゲーションに表示されるのは明示的に参照したオペレーションのみです。その他のページが自動で追加されることはありません。子ナビゲーション要素で参照されたオペレーションも同様です。
  </Accordion>

  <Accordion title="混在ナビゲーション（OpenAPI と MDX ページ）が正しく機能しない">
    OpenAPI のオペレーションと通常のドキュメントページをナビゲーションで組み合わせる場合:

    1. **ファイルの競合**: 同じオペレーションに対して `MDX` ファイルとナビゲーション項目を両方用意することはできません。例えば、`get-users.mdx` がある場合は、ナビゲーションに `"GET /users"` を追加しないでください。オペレーション名と同名のファイルが必要な場合は、エンドポイントの `x-mint` 拡張を使い、href を別の場所を指すように設定してください。
    2. **パス解決**: OpenAPI のオペレーションに一致しないナビゲーション項目は、ファイルパスとして扱われます。`MDX` ファイルが想定どおりの場所に存在することを確認してください。
    3. **大文字小文字の区別**: OpenAPI のオペレーションのマッチングは大文字小文字を区別します。ナビゲーション項目内の HTTP メソッドは大文字で記載してください。
  </Accordion>
</AccordionGroup>
