ワークスペースのルールとメモリを活用して、スタイルガイド、コンポーネント、プロジェクトのコンテキストを理解するドキュメントの専門家として Windsurf を機能させましょう。

<div id="using-windsurf-with-mintlify">
  ## Mintlify で Windsurf を使用する
</div>

Windsurf の Cascade AI アシスタントは、Mintlify のコンポーネントを使って、貴社の基準に沿ったドキュメントを作成するように調整できます。Workspace ルールとメモリは、プロジェクトに関する持続的なコンテキストを提供し、Cascade の提案の一貫性を高めます。

* **Workspace ルール**は、ドキュメントのリポジトリに保存され、チームで共有されます。
* **メモリ**は、時間とともに蓄積される個人別のコンテキストを提供します。

共有ドキュメントの標準化に向けて、Workspace ルールの設定を推奨します。メモリは作業を通じて蓄積できますが、共有されないため、チームメンバー間で一貫性は保たれません。

ドキュメントのリポジトリ内の `.windsurf/rules` ディレクトリに Workspace ルールを作成してください。詳細は Windsurf のドキュメントの [Memories & Rules](https://docs.windsurf.com/windsurf/cascade/memories) を参照してください。

<div id="example-workspace-rule">
  ## ワークスペースルールの例
</div>

このルールは、Mintlify のコンポーネントおよび一般的なテクニカルライティングのベストプラクティスに関するコンテキストを Cascade に提供します。

このサンプルルールはそのまま使用することも、ドキュメントに合わせてカスタマイズすることもできます。

* **執筆標準**: 言語ガイドラインを自社のスタイルガイドに合わせて更新します。
* **コンポーネントパターン**: プロジェクト固有のコンポーネントを追加する、または既存の例を調整します。
* **コード例**: 自社プロダクトの実際の API 呼び出しとレスポンスで汎用的な例を置き換えます。
* **文体とトーンの設定**: 用語、書式、その他のルールを調整します。

ルールはドキュメントリポジトリの `.windsurf/rules` ディレクトリに `.md` ファイルとして保存してください。

````mdx
# Mintlify technical writing rule

## Project context

- これは Mintlify プラットフォーム上のドキュメントプロジェクトです
- YAML フロントマター付きの MDX ファイルを使用します  
- ナビゲーションは `docs.json` で構成します
- テクニカルライティングのベストプラクティスに準拠します

## Writing standards

- 手順は二人称（"you"）で記述します
- 能動態・現在形で記述します
- 手順は前提条件から始めます
- 主要なステップには期待される結果を含めます
- 説明的でキーワードを含む見出しを使用します
- 文章は簡潔かつ有用に保ちます

## Required page structure

すべてのページは次のフロントマターから開始します:

```yaml
---
title: "明確で具体的なタイトル"
description: "SEO とナビゲーションのための簡潔な説明"
---
```

## Mintlify components

### docs.json

- `docs.json` ファイルおよびサイトナビゲーションを作成する際は、[docs.json スキーマ](https://mintlify.com/docs.json)を参照してください

### Callouts

- 補足的で有用な情報には `<Note>`
- 重要な注意点や破壊的変更には `<Warning>`
- ベストプラクティスや専門的助言には `<Tip>`  
- 中立的な補足情報には `<Info>`
- 成功の確認には `<Check>`

### Code examples

- 適切な場合は、完全で実行可能な例を含めます
- 複数言語の例には `<CodeGroup>` を使用します
- すべてのコードブロックに言語タグを指定します
- プレースホルダーではなく現実的なデータを使用します
- API ドキュメントでは `<RequestExample>` と `<ResponseExample>` を使用します

### Procedures

- 連続する手順には `<Steps>` コンポーネントを使用します
- 必要に応じて `<Check>` コンポーネントで検証ステップを含めます
- 複雑な手順は小さなステップに分割します

### Content organization

- プラットフォーム別の内容には `<Tabs>` を使用します
- 段階的に情報を開示するには `<Accordion>` を使用します
- 重要な内容の強調には `<Card>` と `<CardGroup>` を使用します
- 画像は説明的な代替テキスト付きで `<Frame>` コンポーネントにラップします

## API documentation requirements

- すべてのパラメータを `<ParamField>` で記載します
- レスポンス構造は `<ResponseField>` で示します
- 成功例とエラー例の両方を含めます
- 入れ子のオブジェクトプロパティには `<Expandable>` を使用します
- 認証の例は必ず含めます

## Quality standards

- 公開前にすべてのコード例をテストします
- 内部リンクには相対パスを使用します
- すべての画像に代替テキストを含めます
- 見出しの階層が適切か確認します（h2 から開始）
- 既存のパターンとの一貫性を確認します
````

<div id="working-with-cascade">
  ## Cascadeの活用
</div>

ルールを設定したら、Cascadeを使って各種ドキュメント作成作業を効率化できます。詳しくは、Windsurf ドキュメントの[Cascade](https://docs.windsurf.com/windsurf/cascade)を参照してください。

<div id="example-prompts">
  ### プロンプト例
</div>

**新規コンテンツの作成**:

```text wrap
APIの認証方法を説明する新しいページを作成してください。JavaScript、Python、cURL のコード例を含めてください。
```

**既存コンテンツの改善**:

```text wrap
このページを見直し、わかりやすさとコンポーネントの使い方に関する改善点を提案してください。手順をより追いやすくすることに重点を置いてください。
```

**コード例の作成**:

```text wrap
このAPIエンドポイントのエラーハンドリングを示す、完結したコード例を生成してください。現実的なデータを使用し、想定されるレスポンスも含めてください。
```

**一貫性の維持**:

```text wrap
この新しいページがドキュメント標準に準拠しているか確認し、必要な変更点があれば提案してください。
```
