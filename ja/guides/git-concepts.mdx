Git はバージョン管理システムで、ドキュメントの変更を追跡し、チームメンバーと共同作業できます。Git を使えば、プロジェクト内の各ファイルについて、何がいつなぜ変更されたのかを確認できます。必要に応じて、ファイルを以前のバージョンに戻すことも簡単です。

ウェブエディタは、裏側でいくつかの Git 操作を実行します。これらの概念を理解しておくと、ウェブエディタをより効果的に使え、ローカルの開発環境で作業しているチームメンバーとの連携もしやすくなります。

<div id="core-git-concepts">
  ## 主要な Git の概念
</div>

<AccordionGroup>
  <Accordion title="Repository">
    すべてのファイルとその履歴が保存される、ドキュメントのソースです。ウェブエディタはリポジトリに接続して、コンテンツにアクセスし、編集を行います。
  </Accordion>

  <Accordion title="Commit">
    特定の時点での変更を保存したスナップショットです。各コミットには変更内容を説明するメッセージが含まれ、プロジェクト履歴に恒久的な記録が残ります。
  </Accordion>

  <Accordion title="Branch">
    ブランチはリポジトリ内の特定のコミットを指します。公開中のドキュメントはデプロイメントブランチからビルドされます。公開中のドキュメントにはまだ反映されていない変更を含むブランチを、いくつでも作成できます。ブランチの変更を公開中のドキュメントに取り込むには、Pull Request を作成してデプロイメントブランチにマージします。

    ブランチを使うことで、公開中のドキュメントに影響を与えずに作業でき、新機能の安全な検証や、公開前のレビューが可能になります。
  </Accordion>

  <Accordion title="Deployment branch">
    公開中のドキュメントがビルドされる、プロジェクトのメインブランチです。このブランチへの変更は自動的にドキュメントサイトへ公開されます。一般的には `main` と呼ばれますが、任意のブランチをデプロイメントブランチとして設定できます。
  </Accordion>

  <Accordion title="Pull request">
    ブランチ上の変更を公開中のドキュメントにマージするための提案です。変更が公開される前に、レビューや議論を行えます。一般的に PR と呼ばれ、GitLab では Merge Request とも呼ばれます。
  </Accordion>

  <Accordion title="Diff">
    diff（差分）は、ファイルの2つのバージョン間の変更点を示します。Pull Request をレビューする際、diff は追加・削除・修正された箇所を強調表示し、何が変わったかを把握しやすくします。
  </Accordion>
</AccordionGroup>

<div id="how-the-web-editor-uses-git">
  ## ウェブエディタが Git をどのように使うか
</div>

ウェブエディタは [GitHub App](/ja/deploy/github) または [GitLab 連携](/ja/deploy/gitlab) を通じて Git リポジトリに接続し、一般的な Git 操作を自動化します。

次の操作を行うとき:

* **ファイルを開く**: エディタがリポジトリから最新のバージョンを取得し、常に最新のコンテンツで作業できるようにします。
* **変更を加える**: エディタが変更を下書きとして追跡し、保存の準備ができたらコミットにできます。
* **変更を保存する**: エディタが変更をコミットとして作成し、プロジェクト履歴に作業を残します。
* **ブランチを作成する**: エディタがリポジトリに新しいブランチを作成します。リポジトリへのアクセス権があるユーザーなら誰でも利用でき、コラボレーションやレビューが可能になります。
* **デプロイブランチに公開する**: エディタがデプロイブランチに直接コミットしてプッシュし、変更を即時に公開します。
* **その他のブランチで公開する**: エディタがプルリクエストを作成し、デプロイブランチに変更をマージする前に他の人からフィードバックを得られるようにします。

<div id="git-best-practices">
  ## Git のベストプラクティス
</div>

各チームは独自のワークフローや好みを持ちますが、まずは次の一般的なベストプラクティスから始めましょう。

* **わかりやすいコミットメッセージを書く**: 変更点を具体的に、能動的な表現で記述します。
* **意味の伝わるブランチ名を使う**: ブランチ名は、そのブランチで行う作業を説明し、リポジトリのブランチ一覧を見る人にも意味が明確であるべきです。
* **ブランチは目的を絞る**: 各ブランチの変更は、特定のタスクやプロジェクトに焦点を当ててまとめます。
* **マージ後はブランチを削除する**: リポジトリを整理された状態に保つため、不要になったブランチは削除します。
