アクセシブルなドキュメントを作成するとは、ユーザーがどのようにドキュメントへアクセスし、操作するかに関わらず、できるだけ多くの人が利用できるようにするコンテンツ設計を優先することです。

アクセシブルなドキュメントは、すべての人のユーザーエクスペリエンスを向上させます。コンテンツはより明確になり、構造が改善され、より行き来しやすくなります。

このガイドでは、アクセシブルなドキュメントを作成するためのベストプラクティスをいくつか紹介しますが、網羅的ではありません。アクセシビリティは継続的な取り組みと捉えるべきです。技術や標準は時間とともに変化し、それによってドキュメントを改善する新たな機会が生まれます。

<div id="what-is-accessibility">
  ## アクセシビリティとは？
</div>

アクセシビリティ（「accessibility」の最初と最後の文字の間にある文字数に由来して a11y と略されることがあります）とは、できるだけ多くの人が利用できるよう、意図的にウェブサイトやツールを設計・構築することです。恒久的または一時的な障害のある人も、デジタル技術に同等にアクセスできるべきです。また、アクセシビリティに配慮した設計は、モバイルデバイスや低速なネットワークであなたのウェブサイトにアクセスする人など、すべての人にとって有益です。

アクセシブルなドキュメントは、主に [Web Content Accessibility Guidelines (WCAG)](https://www.w3.org/WAI/WCAG22/quickref/) などのウェブアクセシビリティ標準に準拠します。これらのガイドラインは、コンテンツが知覚可能、操作可能、理解可能、かつ堅牢であることを確保するのに役立ちます。

<div id="getting-started-with-accessibility">
  ## アクセシビリティの始め方
</div>

ドキュメントをアクセシブルにするのは継続的な取り組みです。すべてを一度に直す必要はありませんし、一度やって終わりでもありません。

ドキュメントでアクセシビリティの取り組みを始めたばかりなら、まず効果の大きい変更から着手し、段階的に進めるアプローチを検討してください。

<div id="first-steps">
  ### はじめに
</div>

ドキュメントのアクセシビリティを今すぐ改善するための3つのアクション：

1. コンテンツ内のアクセシビリティ上の問題を特定するために、**`mint a11y` を実行**します。
2. すべての画像に**代替テキスト（altテキスト）を追加**します。
3. **見出しの階層を確認**し、各ページに H1 は1つだけ、かつ見出しが順序どおりに並んでいることを確認します。

<div id="plan-your-accessibility-work">
  ### アクセシビリティ対応の計画を立てる
</div>

最適なワークフローは、チームに合ったものです。アクセシビリティ対応に取り組む際の一例を紹介します:

**フェーズ 1: 画像と構造**

- すべての画像の代替テキスト（alt）を見直す。
- リンクテキストを点検し、「click here」のような汎用表現を置き換える。
- ドキュメント全体の見出し階層の問題を修正する。

**フェーズ 2: ナビゲーションとメディア**

- ドキュメントのキーボード操作をテストする。
- スクリーンリーダーでの読み上げをテストする。
- 埋め込み動画にキャプションとトランスクリプトを追加する。
- 色コントラストを確認する。

**フェーズ 3: ワークフローへの組み込み**

- 新しいコンテンツを公開する前に `mint a11y` を実行する。
- コンテンツレビュー工程にアクセシビリティチェックを組み込む。
- インタラクティブ機能を追加する際にキーボード操作をテストする。
- 新しい外部リンクや埋め込みに適切なタイトルと説明が含まれているか確認する。

小さく始め、日常のワークフローにアクセシビリティ対応を組み込むことで、継続可能になります。各改善によって、より多くのユーザーがドキュメントに円滑にアクセスできるようになります。

<div id="structure-your-content">
  ## コンテンツを構造化する
</div>

適切に構造化されたコンテンツは、特に見出しを使ってページ内を移動するスクリーンリーダー利用者や、キーボード操作で閲覧する人にとって、ナビゲーションしやすく理解もしやすくなります。

<div id="use-proper-heading-hierarchy">
  ### 適切な見出し階層を使用する
</div>

各ページには H1 見出しを1つだけ設定します。これはページのフロントマター内の `title:` プロパティで定義されます。見出しは順序を飛ばさずに使用してください。たとえば、H2 から H4 に飛ばさないでください。

```mdx
<!-- 良い例 -->
# ページタイトル (H1)

## メインセクション (H2)

### サブセクション (H3)

### 別のサブセクション (H3)

## 別のメインセクション (H2)

<!-- 悪い例 -->
# ページタイトル (H1)

## メインセクション (H2)

#### サブセクション (H4)

### 別のサブセクション (H3)
```

同じ階層の見出しは固有の名前にしてください。

```mdx
<!-- 良い例 -->
## アクセシビリティのヒント (H2)

### 効果的な代替テキストを書く (H3)

### 適切な色のコントラストを使用する (H3)

<!-- 悪い例 -->
## アクセシビリティのヒント (H2)

### ヒント (H3)

### ヒント (H3)
```


<div id="write-descriptive-link-text">
  ### 意味のあるリンクテキストを書く
</div>

リンクテキストは、遷移先と関連づけられ、内容がわかる表現にしましょう。「click here」や「read more」のような曖昧な表現は避けてください。

```mdx
<!-- 良い例 -->
[ナビゲーションの設定方法](/organize/navigation)について学ぶ。

<!-- 関連性が不明確 -->
[詳細はこちら](/organize/navigation)。
```


<div id="keep-content-scannable">
  ### コンテンツはスキャンしやすく保つ
</div>

- 長い段落は分割する。
- 手順や選択肢にはリストを使う。
- コールアウトで重要な情報を強調する。

<div id="use-proper-table-structure">
  ### 適切な表構造を使用する
</div>

表の使用は必要最小限に留め、列・行の見出しによって意味が成り立つ表形式データにのみ使用してください。

表を使用する場合は、スクリーンリーダーがデータを正しい列と関連付けられるよう、見出しを必ず設定してください:

```mdx
| 機能 | ステータス |
| ------- | ------ |
| 検索  | 有効 |
| アナリティクス | 有効 |
```


<div id="write-descriptive-alt-text">
  ## 説明的な代替テキスト（alt text）を書く
</div>

代替テキストはスクリーンリーダー利用者に画像内容を伝え、画像の読み込みに失敗した場合にも表示されます。ドキュメント内の画像には、画像の内容と掲載目的が分かるように説明する代替テキストを付けてください。代替テキストがあっても、情報の伝達を画像だけに頼るべきではありません。画像が伝える内容は本文でも必ず説明しましょう。

<div id="write-effective-alt-text">
  ### 効果的な代替テキストを書く
</div>

* **具体的に**: 画像が何を示しているのかを説明し、「画像である」といった情報だけにとどめないでください。
* **簡潔に**: 1～2文を目安にしましょう。
* **冗長さを避ける**: スクリーンリーダーは代替テキストが画像に紐づいていることを認識するため、「Image of」で始める必要はありません。ただし、文脈的に重要であれば「Screenshot of」や「Diagram of」といった先頭の説明は含めてください。

```mdx
<!-- 良い例 -->
![3つのアクティブなプロジェクトと2つの保留中の招待が表示されているダッシュボードのスクリーンショット](/images/dashboard.png)

<!-- 良くない例 -->
![ダッシュボードのスクリーンショット](/images/dashboard.png)
```


<div id="add-alt-text-to-images">
  ### 画像に代替テキストを追加する
</div>

Markdown の画像には、角括弧内に代替テキストを記述します：

```mdx
![画像の説明](/path/to/image.png)
```

HTML の画像には `alt` 属性を使用します：

```html
<img
  src="/images/screenshot.png"
  alt="アクセシビリティオプションが有効化された設定パネル。オプションはオレンジ色の矩形で強調表示されています。"
/>
```


<div id="add-titles-to-embedded-content">
  ### 埋め込みコンテンツにタイトルを追加する
</div>

iframe や動画の埋め込みには、内容を説明するタイトルが必要です。

```html
<iframe
  src="https://www.youtube.com/embed/example"
  title="チュートリアル：初めてのドキュメントサイトの構築"
></iframe>
```


<div id="design-for-readability">
  ## 可読性を高めるデザイン
</div>

ビジュアルデザインの選択は、弱視、色覚多様性、その他の視覚障害があるユーザーにとって、ドキュメントのアクセシビリティに影響します。

<div id="ensure-sufficient-color-contrast">
  ### 十分な色コントラストを確保する
</div>

テーマカラーをカスタマイズする場合は、コントラスト比が WCAG の要件を満たしていることを確認してください：

- 本文テキスト：最低 4.5:1 のコントラスト比
- 大きなテキスト：最低 3:1 のコントラスト比
- インタラクティブ要素：最低 3:1 のコントラスト比

ライトモードとダークモードの両方でテストしてください。`mint a11y` コマンドで色コントラストをチェックできます。

<div id="dont-rely-on-color-alone">
  ### 色だけに頼らない
</div>

情報を伝えるために色を使う場合は、テキストラベルやアイコンも併用してください。たとえば、エラーを赤字だけで示さず、エラーアイコンや「エラー」といった文字表示も加えましょう。

<div id="use-clear-concise-language">
  ### 明確で簡潔な言葉を使う
</div>

- 平易な表現で書く。
- 初出時に技術用語を定義する。
- だらだらとした長文を避ける。
- 能動態で書く。

<div id="make-code-examples-accessible">
  ## コード例をアクセシブルにする
</div>

コードブロックは技術ドキュメントの要ですが、スクリーンリーダー利用者が理解できるよう、固有のアクセシビリティ配慮が必要です。基本的には次のガイドラインに従ってください:

- 長いコード例は、より小さく論理的なまとまりに分割する。
- 複雑なロジックには、コード内でコメントを付ける。
- 複雑なアルゴリズムには、テキストによる説明の提供を検討する。
- ファイル構造を示す場合は、ASCIIアートではなく、言語ラベル付きの実際のコードブロックを使用する。

<div id="specify-the-programming-language">
  ### プログラミング言語を指定する
</div>

構文ハイライトのために、必ず使用する言語を指定してください。これにより、スクリーンリーダーがコードの文脈をユーザーに正しく伝えられます。

````mdx
```javascript
function getUserData(id) {
  return fetch(`/api/users/${id}`);
}
```
````


<div id="provide-context-around-code">
  ### コードに関する文脈を提示する
</div>

コードブロックには明確な文脈を付けましょう:

````mdx
次の関数はAPIからユーザーデータを取得します:

```javascript
function getUserData(id) {
  return fetch(`/api/users/${id}`);
}
```

この関数はユーザーオブジェクトに解決されるPromiseを返します。
````


<div id="video-and-multimedia-accessibility">
  ## 動画とマルチメディアのアクセシビリティ
</div>

動画、アニメーション、その他のマルチメディアコンテンツには、すべてのユーザーが内容にアクセスできるように、テキストによる代替手段が必要です。

<div id="add-captions-to-videos">
  ### 動画にキャプションを追加する
</div>

キャプションは、聴覚障害のあるユーザーが動画コンテンツにアクセスできるようにするための重要な手段です。音に配慮が必要な環境や、非ネイティブスピーカーにとっても役立ちます。

- 動画内のすべての発話にキャプションを付ける。
- 関連する効果音もキャプションに含める。
- キャプションが音声と正確に同期していることを確認する。
- 複数人が話す場合は、適切な句読点と話者の明示を行う。

多くの動画ホスティングプラットフォームがキャプションの追加に対応しています。キャプションファイルをアップロードするか、自動生成キャプションをたたき台として活用し、内容の正確性を確認してください。

<div id="provide-transcripts">
  ### 文字起こしを提供する
</div>

文字起こしは、動画コンテンツにアクセスする別の手段です。検索でき、参照しやすく、スクリーンリーダーにも対応しています。

```mdx
<iframe
  src="https://www.youtube.com/embed/example"
  title="チュートリアル：認証の設定"
></iframe>

<Accordion title="動画の文字起こし">
このチュートリアルでは、認証の設定方法について説明します...
</Accordion>
```

動画の近くに文字起こしを配置するか、アクセス用の明確なリンクを用意してください。


<div id="consider-alternatives-to-video-only-content">
  ### 動画だけのコンテンツは避けることを検討する
</div>

重要な情報が動画にしか含まれていない場合は、次を行ってください。

- 同じ情報をテキストでも提供する。
- 説明的なaltテキストを付けた重要なスクリーンショットを含める。
- 同じ内容を扱うチュートリアル記事を作成する。

これにより、動画コンテンツにアクセスできないユーザーでもタスクを完了できます。

<div id="test-your-documentation">
  ## ドキュメントをテストする
</div>

定期的にテストすることで、ユーザーが遭遇する前にアクセシビリティ上の問題を見つけられます。

<div id="check-for-accessibility-issues-with-mint-a11y">
  ### `mint a11y` でアクセシビリティの問題をチェックする
</div>

`mint a11y` CLI コマンドを使って、ドキュメントに一般的なアクセシビリティ上の問題がないか自動スキャンします。

```bash
mint a11y
```

このコマンドは次の点をチェックします:

* 画像の alt テキストの欠如
* 見出し階層の不適切さ
* カラーコントラストの不足

スキャンが完了したら、報告された問題を確認し、コンテンツを修正してください。修正内容を確認するために、コマンドをもう一度実行してください。


<div id="basic-keyboard-navigation-test">
  ### 基本的なキーボード操作のテスト
</div>

キーボードのみでドキュメント内を操作します：

1. <kbd>Tab</kbd> を押して、インタラクティブ要素間を順方向に移動します。
2. <kbd>Shift</kbd> + <kbd>Tab</kbd> を押して、逆方向に移動します。
3. <kbd>Enter</kbd> を押して、リンクやボタンを実行します。
4. すべてのインタラクティブ要素に到達でき、フォーカスインジケーターが視認できることを確認します。

<div id="go-deeper-with-accessibility-testing">
  ### アクセシビリティテストをさらに深める
</div>

より網羅的なテストのために:

- **スクリーンリーダー**: [NVDA（Windows）](https://www.nvaccess.org/) や [VoiceOver（Mac）](https://www.apple.com/accessibility/voiceover/) でテストします。
- **ブラウザー拡張機能**: [axe DevTools](https://www.deque.com/axe/browser-extensions/) や [WAVE](https://wave.webaim.org/extension/) を導入して、ページの問題をスキャンします。
- **WCAG ガイドライン**: 詳細な基準は [Web Content Accessibility Guidelines](https://www.w3.org/WAI/WCAG22/quickref/) を参照してください。
