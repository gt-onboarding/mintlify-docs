Git 是一种版本控制系统，用于跟踪文档的更改并与团队成员协作。借助 Git，你可以查看项目中每个文件的更改内容、时间以及原因；在需要时也能轻松恢复到文件的早期版本。

网页编辑器会在后台执行一些 Git 操作。理解这些概念将帮助你更高效地使用网页编辑器，并与在本地开发环境中工作的团队成员协作。

<div id="core-git-concepts">
  ## 核心 Git 概念
</div>

<AccordionGroup>
  <Accordion title="Repository">
    你的文档源代码所在地，存放所有文件及其变更历史。网页编辑器会连接到你的仓库以访问和修改内容。
  </Accordion>

  <Accordion title="Commit">
    在特定时间点保存的一次更改快照。每个提交都包含描述更改的消息，并在项目历史中创建永久记录。
  </Accordion>

  <Accordion title="Branch">
    分支指向仓库中的某个特定提交。你的线上文档是从部署分支构建的。你可以拥有任意数量的其他分支，这些分支的更改尚未发布到线上文档。若要将某个分支的更改纳入线上文档，可以通过拉取请求将其合并到部署分支。

    使用分支可以在不影响线上文档的情况下进行修改，安全地尝试新功能，并在发布前完成评审。
  </Accordion>

  <Accordion title="Deployment branch">
    项目的主分支，线上文档内容从该分支构建。对此分支的更改会自动发布到你的文档站点。通常称为 `main`，但你可以将任意分支设为部署分支。
  </Accordion>

  <Accordion title="Pull request">
    用于提议将某个分支上的更改合并到线上文档的方式。允许在更改上线前进行评审与讨论。通常称为 PR，在 GitLab 中也称为 Merge Request。
  </Accordion>

  <Accordion title="Diff">
    diff（差异）用于显示文件两个版本之间的更改。在审阅拉取请求时，diff 会高亮新增、删除或修改的内容，便于识别变更。
  </Accordion>
</AccordionGroup>

<div id="how-the-web-editor-uses-git">
  ## 网页编辑器如何使用 Git
</div>

网页编辑器通过 [GitHub 应用](/zh/deploy/github) 或 [GitLab 集成](/zh/deploy/gitlab) 连接到你的 Git 仓库，并自动执行常见的 Git 操作。

当你：

* **打开文件**：编辑器会从仓库获取最新版本，确保你始终处理的是最新内容。
* **进行更改**：编辑器会将你的更改作为草稿进行跟踪，在你准备保存时可转换为一次提交（commit）。
* **保存更改**：编辑器会根据你的更改创建一次提交，将你的工作保存在项目历史中。
* **创建分支**：编辑器会在你的仓库中创建一个新分支，任何拥有仓库访问权限的人都可以使用该分支进行协作与审查。
* **在部署分支上发布**：编辑器会直接向你的部署分支提交并推送，从而立即发布更改。
* **在其他分支上发布**：编辑器会创建一个拉取请求（pull request），让你在将更改合并到部署分支之前获取他人的反馈。

<div id="git-best-practices">
  ## Git 最佳实践
</div>

每个团队都会形成自己的工作流程和偏好，但以下是一些通用的最佳实践，帮助你快速上手。

* **编写清晰的提交信息**：使用主动语态，具体说明改动内容。
* **使用清晰的分支命名**：分支名应能说明正在进行的工作，并且对查看你仓库分支的人一目了然。
* **保持分支聚焦**：让分支上的更改专注于某个特定任务或项目。
* **合并后删除分支**：不再需要时删除分支，保持仓库整洁。
