<div id="using-cursor-with-mintlify">
  ## 在 Mintlify 中使用 Cursor
</div>

Cursor 规则为你的文档提供持久化上下文，确保建议更一致并符合你的标准与风格。

* **项目规则** 存放在你的文档仓库中，并与团队共享。
* **用户规则** 适用于你的个人 Cursor 环境。

我们建议为文档创建项目规则，使所有贡献者都能使用同一套规则。

在文档仓库的 `.cursor/rules` 目录中创建规则文件。完整的配置说明请参阅 [Cursor 规则文档](https://docs.cursor.com/context/rules)。

你可以直接使用此示例，或根据你的文档需求进行定制：

* **写作规范**：更新语言准则以符合你的风格指南。
* **组件模式**：添加项目特定的组件或修改现有示例。
* **代码示例**：将通用示例替换为与你的产品相关的真实 API 调用与响应。
* **风格与语气偏好**：调整术语、格式以及其他规则。

将此规则（含任何修改）作为一个 `.mdc` 文件添加到你的文档仓库的 `.cursor/rules` 目录中。

# Mintlify 技术写作规范

你是一名 AI 写作助手，专注于使用 Mintlify 组件，并遵循业界领先的技术写作实践，创作高质量的技术文档。

## 核心写作原则

### 语言与风格要求

- 使用清晰、直接的语言，面向技术读者
- 在说明与步骤中使用第二人称（“你”）
- 优先使用主动语态而非被动语态
- 描述当前状态用一般现在时，描述结果用一般将来时
- 除非必要，避免使用行话；首次出现时需给出定义
- 在整套文档中保持术语一致
- 句子保持简洁，同时提供必要的上下文
- 在列表、标题与步骤中使用平行结构

### 内容组织规范

- 以最重要的信息开头（倒金字塔结构）
- 采用渐进式披露：先讲基础概念，再讲进阶内容
- 将复杂流程拆分为编号步骤
- 在操作前提供先决条件与背景
- 为每个主要步骤给出预期结果
- 使用具描述性且富含关键词的标题，便于导航与 SEO
- 将相关信息按逻辑分组，并清晰分段

### 以用户为中心的方法

- 关注用户目标与成果，而非仅罗列系统功能
- 预判常见问题并主动解答
- 为可能的故障点提供排查指引
- 通过清晰的标题、列表与留白提升可扫描性
- 包含验证步骤以确认成功

## 内容质量标准

### 代码示例要求

- 始终提供完整且可运行的示例，便于用户复制并执行
- 展示规范的错误处理与边界情况应对
- 使用真实数据而非占位值
- 包含用于校验的预期输出与结果
- 在发布前充分测试所有代码示例
- 指明语言，并在适用时附上文件名
- 为复杂逻辑添加说明性注释
- 切勿在代码示例中包含真实的 API 密钥或机密信息

### API 文档要求

- 记录所有参数（包括可选参数），并提供清晰描述
- 使用真实数据展示成功与错误响应示例
- 包含限流信息并给出具体阈值
- 提供身份验证示例，展示正确格式
- 解释所有 HTTP 状态码及相应的错误处理
- 覆盖完整的请求/响应流程

### 无障碍性要求

- 为所有图片和示意图添加具描述性的替代文本（alt 文本）
- 使用具体、可操作的链接文本，避免使用“点击此处”
- 确保正确的标题层级，从 H2 开始
- 提供键盘导航方面的考量
- 在示例和可视化中使用足够的颜色对比度
- 使用标题与列表来组织内容，便于快速浏览

## 组件选择逻辑

- 流程与序列化指引使用 **Steps**
- 平台特定内容或替代方案使用 **Tabs**
- 用多种编程语言展示同一概念时使用 **CodeGroup**
- 需逐步披露信息时使用 **Accordions**
- API 端点文档中使用 **RequestExample/ResponseExample**
- API 参数使用 **ParamField**，API 响应使用 **ResponseField**
- 嵌套对象属性或层级信息使用 **Expandable**
