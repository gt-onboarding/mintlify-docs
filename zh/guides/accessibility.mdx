当你面向无障碍创作文档时，会优先考虑内容设计，使尽可能多的人都能使用你的文档，而不受其访问和交互方式的限制。

无障碍文档能提升所有人的使用体验。你的内容会更清晰、结构更合理、导航更便捷。

本指南提供了一些创建无障碍文档的最佳实践，但并不穷尽所有内容。你应将无障碍视为一个持续的过程。技术和标准会随时间演进，也会带来改进文档的新机会。

<div id="what-is-accessibility">
  ## 什么是无障碍？
</div>

无障碍（有时缩写为 a11y，表示“accessibility”首尾字母之间相隔的 11 个字母）是指有意识地设计和构建尽可能多的人都能使用的网站和工具。具有临时或永久性障碍的人应当享有与他人同等水平的数字技术可达性。面向无障碍的设计也能惠及所有人，包括在移动设备或慢速网络上访问你的网站的用户。

无障碍文档应遵循网页无障碍标准，主要是 [Web Content Accessibility Guidelines（WCAG）](https://www.w3.org/WAI/WCAG22/quickref/)。这些指南有助于确保你的内容可感知、可操作、可理解且具备鲁棒性。

<div id="getting-started-with-accessibility">
  ## 无障碍入门
</div>

让你的文档具备无障碍性是一个持续的过程。你不必一次性解决所有问题，也不能一次完成后就结束。

如果你刚开始在文档中落实无障碍实践，建议采用分阶段的方法，从影响最大的改动着手，逐步推进。

<div id="first-steps">
  ### 入门
</div>

以下是你现在就可以做的三件事，来提升文档的无障碍性：

1. **运行 `mint a11y`**，以识别内容中的无障碍问题。
2. **为所有图像添加 alt 文本**。
3. **检查标题层级**，确保每页仅有一个 H1，且标题按顺序递进。

<div id="plan-your-accessibility-work">
  ### 规划你的无障碍工作
</div>

最好的工作流程是最适合你团队的那一个。以下是你可以开展无障碍工作的一个思路：

**阶段 1：图像与结构**

- 检查所有图像是否包含具有描述性的 alt 文本。
- 审核链接文本，替换诸如“点击这里”之类的通用短语。
- 修复文档中的标题层级问题。

**阶段 2：导航与媒体**

- 测试文档的键盘可导航性。
- 测试屏幕阅读器的兼容性与支持情况。
- 为嵌入的视频添加字幕和文字稿。
- 检查颜色对比度。

**阶段 3：将其纳入你的工作流程**

- 在发布新内容前运行 `mint a11y`。
- 在内容评审流程中加入无障碍检查。
- 添加交互功能时测试键盘导航。
- 确认新的外部链接和嵌入内容包含恰当的标题与描述。

从小处着手，并将无障碍纳入日常工作流程，才能长期坚持。每一次改进都能帮助更多用户顺利获取你的文档。

<div id="structure-your-content">
  ## 结构化你的内容
</div>

结构清晰的内容更易于导航和理解，尤其有助于依靠标题在页面间移动的屏幕阅读器用户，以及使用键盘进行导航的用户。

<div id="use-proper-heading-hierarchy">
  ### 使用正确的标题层级
</div>

每个页面应只有一个 H1 标题，它由页面的 frontmatter 中的 `title:` 属性定义。按顺序使用其余标题，避免跳级。例如，不要从 H2 直接跳到 H4。

```mdx
<!-- 正确示例 -->
# 页面标题 (H1)

## 主要章节 (H2)

### 子章节 (H3)

### 另一个子章节 (H3)

## 另一个主要章节 (H2)

<!-- 错误示例 -->
# 页面标题 (H1)

## 主要章节 (H2)

#### 子章节 (H4)

### 另一个子章节 (H3)
```

同一级别的标题应具有唯一的名称。

```mdx
<!-- 正确 -->
## 无障碍访问提示 (H2)

### 编写有效的替代文本 (H3)

### 使用适当的颜色对比度 (H3)

<!-- 错误 -->
## 无障碍访问提示 (H2)

### 提示 (H3)

### 提示 (H3)
```


<div id="write-descriptive-link-text">
  ### 使用具描述性的链接文本
</div>

链接文本应当清晰传达含义，并与目标内容紧密相关。避免使用诸如“点击此处”或“了解更多”之类的模糊短语。

```mdx
<!-- 好的示例 -->
了解如何[配置导航](/organize/navigation)。

<!-- 链接文本与目标页面关系不明确 -->
[了解更多](/organize/navigation)。
```


<div id="keep-content-scannable">
  ### 保持内容便于快速扫描
</div>

- 拆分长段落。
- 用列表呈现步骤和选项。
- 通过提示框突出关键信息。

<div id="use-proper-table-structure">
  ### 正确使用表格结构
</div>

谨慎使用表格，仅在展示依赖行列标题传达含义的表格数据时使用。

使用表格时，应添加表头，便于屏幕阅读器将数据与正确的列对应：

```mdx
| 功能 | 状态 |
| ------- | ------ |
| 搜索  | 已启用 |
| 分析 | 已启用 |
```


<div id="write-descriptive-alt-text">
  ## 撰写清晰的替代文字（alt text）
</div>

替代文字有助于屏幕阅读器用户访问图像，并会在图像加载失败时显示。文档中的图像应包含能描述图像并说明其用途或存在原因的替代文字。即使提供了替代文字，也不应仅依赖图像来传达信息。请确保你的正文内容也清楚表达图像所传达的信息。

<div id="write-effective-alt-text">
  ### 撰写有效的替代文字（alt text）
</div>

* **具体明确**：描述图像所呈现的内容，而不是仅说明这是一张图片。
* **简洁精炼**：控制在一到两句话。
* **避免冗余**：不要以“Image of”开头，因为屏幕阅读器已知替代文字对应的是图像。但如果这类语境对理解很重要，可以包含诸如“Screenshot of”或“Diagram of”之类的描述。

```mdx
<!-- 好的示例 -->
![仪表板截图，显示三个活跃项目和两个待处理邀请](/images/dashboard.png)

<!-- 不够有用的示例 -->
![仪表板截图](/images/dashboard.png)
```


<div id="add-alt-text-to-images">
  ### 为图像添加替代文本（alt text）
</div>

对于 Markdown 图像，请在方括号中填写替代文本（alt text）：

```mdx
![图片描述](/path/to/image.png)
```

对于 HTML 图片，请使用 `alt` 属性：

```html
<img
  src="/images/screenshot.png"
  alt="设置面板，已启用无障碍功能选项。选项以橙色矩形框突出显示。"
/>
```


<div id="add-titles-to-embedded-content">
  ### 为嵌入内容添加标题
</div>

Iframes 和视频嵌入需要添加具有描述性的标题：

```html
<iframe
  src="https://www.youtube.com/embed/example"
  title="教程:设置您的第一个文档站点"
></iframe>
```


<div id="design-for-readability">
  ## 以可读性为核心进行设计
</div>

视觉设计的取舍会影响你的文档对低视力、色觉缺陷及其他视觉障碍用户的可访问性。

<div id="ensure-sufficient-color-contrast">
  ### 确保足够的颜色对比度
</div>

如果你自定义了主题颜色，请确认对比度符合 WCAG 要求：

- 正文：至少 4.5:1 的对比度
- 大号文本：至少 3:1 的对比度
- 交互元素：至少 3:1 的对比度

请同时测试浅色和深色模式。`mint a11y` 命令可用于检查颜色对比度。

<div id="dont-rely-on-color-alone">
  ### 不要仅依赖颜色
</div>

如果使用颜色来传达信息，请同时提供文本标签或图标。例如，不要仅用红色文字标注错误；应加入错误图标或加上“Error”字样。

<div id="use-clear-concise-language">
  ### 使用清晰、简洁的语言
</div>

- 使用通俗易懂的表达。
- 第一次出现时解释技术术语。
- 避免冗长的长句。
- 使用主动语态。

<div id="make-code-examples-accessible">
  ## 提升代码示例的无障碍性
</div>

代码块是技术文档的核心组成，但需要特别的无障碍设计，确保屏幕阅读器用户也能理解。一般请遵循以下指南：

- 将较长的代码示例拆分为更小、逻辑清晰的部分。
- 在代码中为复杂逻辑添加注释。
- 为复杂算法考虑提供文字说明。
- 展示文件结构时，使用带有语言标注的真实代码块，而非 ASCII 图示。

<div id="specify-the-programming-language">
  ### 指定编程语言
</div>

务必声明用于语法高亮的语言。这有助于屏幕阅读器向用户说明代码的上下文：

````mdx
```javascript
function getUserData(id) {
  return fetch(`/api/users/${id}`);
}
```
````


<div id="provide-context-around-code">
  ### 为代码提供上下文
</div>

为代码块提供清晰的背景说明：

````mdx
以下函数从 API 获取用户数据：

```javascript
function getUserData(id) {
  return fetch(`/api/users/${id}`);
}
```

此函数返回一个 Promise，该 Promise 会解析为用户对象。
````


<div id="video-and-multimedia-accessibility">
  ## 视频与多媒体的可访问性
</div>

视频、动画和其他多媒体内容需要提供文本替代，以便所有用户都能获取其中的信息。

<div id="add-captions-to-videos">
  ### 为视频添加字幕
</div>

字幕可让失聪或听力受损的用户无障碍访问视频内容。它们也能帮助处于对声音敏感的环境中的用户以及非母语使用者：

- 为视频中的所有口语内容添加字幕。
- 在字幕中包含相关的音效说明。
- 确保字幕与音频同步。
- 当多人发言时，使用恰当的标点并标注说话者。

大多数视频托管平台都支持添加字幕。你可以上传字幕文件，或先使用自动生成的字幕，再对其进行准确性审核。

<div id="provide-transcripts">
  ### 提供文字稿
</div>

文字稿为获取视频内容提供了另一种方式。它们可搜索、便于引用，并且对屏幕阅读器更友好：

```mdx
<iframe
  src="https://www.youtube.com/embed/example"
  title="教程：设置身份验证"
></iframe>

<Accordion title="视频文字稿">
在本教程中，我们将逐步介绍如何设置身份验证...
</Accordion>
```

将文字稿放在视频附近，或提供明显的访问链接。


<div id="consider-alternatives-to-video-only-content">
  ### 考虑视频内容的替代方案
</div>

如果关键信息只出现在视频中：

- 以文本形式提供同样的信息。
- 附上关键截图，并配有描述性替代文本（alt 文本）。
- 编写覆盖相同内容的图文教程。

这样可以确保无法访问视频的用户仍然能够完成他们的任务。

<div id="test-your-documentation">
  ## 测试你的文档
</div>

通过定期测试，可在用户遇到问题之前发现无障碍问题。

<div id="check-for-accessibility-issues-with-mint-a11y">
  ### 使用 `mint a11y` 检查可访问性问题
</div>

使用 `mint a11y` CLI 命令自动扫描文档，检测常见的可访问性问题：

```bash
mint a11y
```

该命令会检查：

* 图像缺少 alt 文本
* 标题层级不正确
* 颜色对比度不足

扫描完成后，查看报告的问题并在内容中修复。再次运行该命令以验证修复结果。


<div id="basic-keyboard-navigation-test">
  ### 基本键盘导航测试
</div>

仅使用键盘在文档中进行导航：

1. 按 <kbd>Tab</kbd> 在交互式元素间向前移动。
2. 按 <kbd>Shift</kbd> + <kbd>Tab</kbd> 向后移动。
3. 按 <kbd>Enter</kbd> 激活链接和按钮。
4. 确认所有交互式元素均可访问，并且具有可见的焦点高亮/指示。

<div id="go-deeper-with-accessibility-testing">
  ### 深入进行无障碍测试
</div>

如需更全面的测试：

- **屏幕阅读器**：使用 [NVDA（Windows）](https://www.nvaccess.org/) 或 [VoiceOver（Mac）](https://www.apple.com/accessibility/voiceover/) 进行测试。
- **浏览器扩展**：安装 [axe DevTools](https://www.deque.com/axe/browser-extensions/) 或 [WAVE](https://wave.webaim.org/extension/)，对页面进行问题扫描。
- **WCAG 指南**：查阅 [Web 内容无障碍指南](https://www.w3.org/WAI/WCAG22/quickref/)，了解详细标准。
