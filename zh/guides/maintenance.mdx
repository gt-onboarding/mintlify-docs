<div id="automate-what-you-can">
  ## 能自动化的就自动化
</div>

在可行的地方引入自动化，例如：

* **跟踪陈旧内容：** 运行脚本标记过去三个月未更新的重要文档。它们仍然准确吗？
* **自动更新文档：** 构建工作流，在代码合并时通过 [agent API](/zh/guides/automate-agent) 自动更新文档。
* **用 linter 执行规范：** 使用 [Vale](http://Vale.sh) 或 [CI 检查](/zh/deploy/ci)，在每个 pull request 中自动发现格式问题、写作风格偏差或缺失的元数据。

<div id="set-up-a-review-process">
  ## 建立审核流程
</div>

文档可能永远无法尽善尽美，这没关系。你应设定一个可接受的标准，只要文档可用且有用即可。

在效率与质量之间取得平衡：

* **优先关注高影响的文档。** 并非每个页面都需要定期更新。务必定期审核最重要的页面，确保其准确且相关。
* **善用社区力量。** 如果你的文档是开源的，鼓励用户通过 pull request 提交问题或修复。这样既能建立信任，也能保持内容常新。

<div id="know-when-to-rewrite">
  ## 何时该重写
</div>

随着时间推移，文档难免会堆积各种注意事项和权宜之计。当逐步修补带来的困惑多于清晰时，全面重构可能是更好的选择。

* **规划定期重置。** 一次大规模清理，尤其是在最佳实践或产品本身发生重大演进时，能为你的团队和用户节省时间。
* **从结构化审查开始。** 在重写之前，先访谈支持团队、分析用户反馈，并梳理出缺失、误导或冗余的内容。
* **分阶段冲刺完成重写。** 全面重构不必一蹴而就，优先处理影响最大的部分。

<div id="wrong-docs-can-be-worse-than-no-docs">
  ## 错误的文档可能比没有文档更糟
</div>

过时或带有误导性的文档会浪费用户时间并削弱信任。如果某个页面在短期内完全不准确且无法修复，通常最好直接将其移除。与其提供错误的信息，用户更愿意看到更少但正确的信息。
