要在使用 AWS Route 53 和 CloudFront 的情况下，将文档托管在以 `/docs` 为子路径的地址，需要在 DNS 提供商处配置记录指向你的 CloudFront 分发。


<div id="proxies-with-vercel-deployments">
  ## 在 Vercel 部署中使用代理
</div>

如果你在 Vercel 部署中使用 AWS CloudFront 作为代理，必须正确配置 CloudFront，避免干扰 Vercel 的域名验证和 SSL 证书签发。

错误的 CloudFront 配置可能会导致 Vercel 无法签发 Let's Encrypt SSL 证书，并引发域名验证失败。

<div id="required-path-allowlist">
  ### 必需的路径白名单
</div>

CloudFront 必须允许以下特定路径的流量通过，且不得缓存或阻止：

- `/.well-known/acme-challenge/*` - 用于 Let's Encrypt 证书验证
- `/.well-known/vercel/*` - 用于 Vercel 域名验证

这些路径应配置为绕过 CloudFront 缓存，并直接透传到源站（Origin）。

<div id="header-forwarding-requirements">
  ### 标头转发要求
</div>

你必须创建一个自定义的源站请求策略，以正确转发 `HOST` 标头和客户端 IP 信息。这对于 Vercel 的验证流程至关重要。

1. 创建一个名为 `VercelCloudFrontProxy` 的自定义源站请求策略。
2. 将 `Origin` 和 `CloudFront-Viewer-Address` 标头纳入转发。

你必须在源站请求策略或缓存策略的标头配置中包含 `CloudFront-Viewer-Address`，以便将该标头转发到源站。

<div id="create-cloudfront-distribution">
  ## 创建 CloudFront 分配
</div>

1. 在 AWS 控制台中进入 [CloudFront](https://aws.amazon.com/cloudfront)。
2. 选择 **Create distribution**。

<Frame>
    ![CloudFront 分配页面，突出显示 “Create distribution” 按钮。](/images/cloudfront/create-distribution.png)
  </Frame>

3. 在 Origin domain 中输入 `[SUBDOMAIN].mintlify.dev`，其中 `[SUBDOMAIN]` 是你项目的唯一子域名。

<Frame>
    ![CloudFront “Create distribution” 页面，显示将 “acme.mintlify.dev” 作为 Origin domain。](/images/cloudfront/origin-name.png)
  </Frame>

4. 在 “Web Application Firewall (WAF)” 中，启用安全防护。

<Frame>
    ![Web Application Firewall (WAF) 选项中，已选择 “Enable security protections”。](/images/cloudfront/enable-security-protections.png)
  </Frame>

5. 其余设置保持默认。
6. 选择 **Create distribution**。

<div id="add-default-origin">
  ## 添加默认 Origin
</div>

1. 创建分发后，前往“Origins”选项卡。

<Frame>
    ![突出显示“Origins”选项卡的 CloudFront 分发。](/images/cloudfront/origins.png)
  </Frame>

2. 找到与主域名对应的预发布环境 URL。其形式会因落地页的托管方式而异。例如，Mintlify 的预发布环境 URL 是 [mintlify-landing-page.vercel.app](https://mintlify-landing-page.vercel.app)。

<Info>
  如果你的落地页托管在 Webflow 上，请使用 Webflow 的预发布环境 URL，通常以 `.webflow.io` 结尾。

  如果你使用 Vercel，请使用每个项目自带的 `.vercel.app` 域名。
</Info>

3. 新建一个 Origin，并将你的预发布环境 URL 填入“Origin domain”。

<Frame>
    ![CloudFront 的“Create origin”页面，突出显示“Origin domain”输入字段。](/images/cloudfront/default-origin.png)
  </Frame>

此时，你应当有两个 Origin：一个是 `[SUBDOMAIN].mintlify.app`，另一个是你的预发布环境 URL。

<Frame>
  ![CloudFront 的“Origins”页面显示两个 origin：一个用于 `mintlify`，另一个用于 `mintlify-landing-page`。](/images/cloudfront/final-origins.png)
</Frame>

<div id="set-behaviors">
  ## 设置行为
</div>

CloudFront 的行为用于控制子路径逻辑。总体上，我们希望实现以下逻辑：

- **如果用户访问 /docs**，跳转到 `[SUBDOMAIN].mintlify.dev`。
- **如果用户访问其他任意页面**，保持当前落地页。

1. 在 CloudFront 的 “Behaviors” 选项卡中进行操作。

<Frame>
	![高亮显示的 CloudFront “Behaviors” 选项卡。](/images/cloudfront/behaviors.png)
</Frame>

2. 点击 **Create behavior** 按钮，创建如下行为。

<div id="well-known">
  ### `/.well-known/*`
</div>

为 Vercel 域名验证路径创建一个 **Path pattern** 为 `/.well-known/*` 的行为，并将 **Origin and origin groups** 指向你的文档站点 URL。

在 “Cache policy” 中选择 **CachingDisabled**，以确保这些验证请求直通且不被缓存。

<Frame>
  ![CloudFront “Create behavior” 页面，其中 “Path pattern” 为 “/.well-known/*”，且 “Origin and origin groups” 指向预发布环境的 URL。](/images/cloudfront/well-known-policy.png)
</Frame>

<Info>
如果 `/.well-known/*` 过于宽泛，针对 Vercel 至少可细化为 2 个行为：
  - `/.well-known/vercel/*` - Vercel 域名验证所需
  - `/.well-known/acme-challenge/*` - Let's Encrypt 证书验证所需
</Info>

<div id="docs">
  ### `/docs`
</div>

创建一个 **Path pattern** 为 `/docs` 的行为，并将 **Origin and origin groups** 指向 `.mintlify.dev` 的 URL（本例为 `acme.mintlify.dev`）。

- 将“Cache policy”设置为 **CachingOptimized**。
- 在“Origin request policy”中，创建一个名为 **VercelCloudFrontProxy** 的源请求策略，用于转发 `Origin` 和 `CloudFront-Viewer-Address` 头。

<Frame>
	![CloudFront 创建源请求策略页面，用于转发 `Origin` 和 `CloudFront-Viewer-Address` 头](/images/cloudfront/origin-request-policy.png)
</Frame>

- 将 Viewer Protocol Policy 设置为 **Redirect HTTP to HTTPS**

<Frame>
  ![CloudFront “Create behavior” 页面，“Path pattern”为“/docs/*”，且“Origin and origin groups”指向 `acme.mintlify.dev` URL。](/images/cloudfront/behavior-1.png)
</Frame>

<div id="docs">
  ### `/docs/*`
</div>

创建一个 **Path pattern** 为 `/docs/*`，且 **Origin and origin groups** 指向同一 `.mintlify.dev` URL 的行为（Behavior）。

这些设置应与 `/docs` 完全一致，唯独 **Path pattern** 例外。

- 将“Cache policy”设置为 **CachingOptimized**
- 将“Origin request policy”设置为 **VercelCloudFrontProxy**
- 将“Viewer protocol policy”设置为 **Redirect HTTP to HTTPS**

<div id="default">
  ### `Default (*)`
</div>

最后，我们将编辑 `Default (*)` 行为。

<Frame>
  ![选择了“Default (*)”行为并突出显示 Edit 按钮的 CloudFront 分配。](/images/cloudfront/default-behavior-1.png)
</Frame>

1. 将默认行为的 **Origin and origin groups** 修改为预发布环境的 URL（在本例中为 `mintlify-landing-page.vercel.app`）。

<Frame>
  ![CloudFront 的 “Edit behavior” 页面，已高亮显示 “Origin and origin groups” 输入字段。](/images/cloudfront/default-behavior-2.png)
</Frame>

2. 选择 **Save changes**。

<div id="check-behaviors-are-set-up-correctly">
  ### 检查行为是否已正确设置
</div>

如果你按照上述步骤配置，Behaviors 应如下所示：

<Frame>
  ![CloudFront「Behaviors」页面，包含 4 个行为：`/docs/*`、`/docs`、`Default` 和 `/.well-known/*`。](/images/cloudfront/all-behaviors.png)
</Frame>

<div id="preview-distribution">
  ## 预览分发
</div>

现在你可以前往“General”选项卡并访问“Distribution domain name”的 URL，测试分发是否已正确配置。

<Frame>
  ![CloudFront “General” 选项卡，其中 “Distribution domain name” URL 被高亮显示。](/images/cloudfront/preview-distribution.png)
</Frame>

所有页面应重定向到你的主落地页；如果在 URL 末尾追加 `/docs`，应跳转到你的 Mintlify 文档实例。

<div id="connect-with-route53">
  ## 通过 Route53 连接
</div>

现在，我们将把 CloudFront 分配的功能应用到你的主域名上。

<Note>
  在本节中，你也可以参考 AWS 的官方指南：[将 Amazon Route 53 配置为将流量路由到 CloudFront 分配](https://docs.aws.amazon.com/Route53/latest/DeveloperGuide/routing-to-cloudfront-distribution.html#routing-to-cloudfront-distribution-config)
</Note>

1. 在 AWS 控制台中打开 [Route53](https://aws.amazon.com/route53)。
2. 前往主域名的“Hosted zone”。
3. 选择 **Create record**。

<Frame>
  ![Route 53 “Records” 页面，突出显示了 “Create record” 按钮。](/images/cloudfront/route53-create-record.png)
</Frame>

4. 打开 `Alias`，然后在 **Route traffic to** 中选择 `Alias to CloudFront distribution` 选项。

<Frame>
  ![Route 53 “Create record” 页面，突出显示了 “Alias” 开关和 “Route traffic to” 菜单。](/images/cloudfront/create-record-alias.png)
</Frame>

5. 选择 **Create records**。

<Note>
  如果当前存在 A 记录，你可能需要先将其移除。
</Note>

你的文档现在已经在主域名的 `/docs` 上线。
