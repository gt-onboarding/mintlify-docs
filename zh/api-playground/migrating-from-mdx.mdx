---
title: "将 MDX API 页面迁移到 OpenAPI 导航"
sidebarTitle: "从 MDX 迁移"
description: "从单个 MDX 端点页面迁移为基于 OpenAPI 的自动生成，并具备灵活的导航"
icon: "arrow-big-right-dash"
---

如果您目前为 API 端点使用单独的 `MDX` 页面，您可以迁移为根据 OpenAPI 规范自动生成页面，同时保留对单个页面的自定义能力。这样可以减少需要维护的文件数量，并提升 API 文档的一致性。

您可以在 OpenAPI 规范中为每个端点定义元数据和内容，并在导航中将端点组织到所需位置。

<div id="cli-migration">
  ## CLI 迁移
</div>

`mint migrate-mdx` 命令是从 MDX 端点页面迁移到自动生成页面的推荐方式。

该命令将：

- 解析你的 `docs.json` 导航结构。
- 识别会生成 OpenAPI 端点页面的 MDX 页面。
- 从 MDX 文件中提取内容并将其移至 OpenAPI 规范中的 `x-mint` 扩展。
- 更新你的 `docs.json`，使其直接引用 OpenAPI 端点而非 MDX 文件。
- 删除原始的 MDX 端点文件。

<Info>
  如果你已为某个端点定义了 `x-mint`，并且也有一个包含该端点内容的 MDX 页面，则该 MDX 页面中的内容将覆盖现有的 `x-mint` 设置。

  如果你为同一端点提供了多个内容不同的 MDX 页面，脚本将使用在你的 `docs.json` 中最后出现的页面内容。

  迁移工具不支持在应用之前预览更改。
</Info>

<Steps>
  <Step title="准备你的 OpenAPI 规范。">
    确保你的 OpenAPI 规范有效，并包含你希望记录的所有端点。

    需要迁移的任何 MDX 页面必须在 frontmatter 中使用 `openapi:` 引用某个端点。

    <Tip>
      使用 [Swagger Editor](https://editor.swagger.io/) 或 [Mint CLI](https://www.npmjs.com/package/mint) 验证你的 OpenAPI 文件。
    </Tip>
  </Step>

  <Step title="安装 Mint 命令行工具">
    如有需要，安装或更新 [Mint CLI](/zh/installation)。
  </Step>

  <Step title="运行迁移命令。">
    ```bash
    mint migrate-mdx
    ```
  </Step>
</Steps>

<div id="manual-migration-steps">
  ## 手动迁移步骤
</div>

<Steps>
  <Step title="准备 OpenAPI 规范。">
    确保你的 OpenAPI 规范有效，并包含你希望记录的所有端点。

    对于需要自定义元数据或内容的端点，请为该端点添加 `x-mint` 扩展。详情参见 [x-mint 扩展](/zh/api-playground/openapi-setup#x-mint-extension)。

    对于需要从文档中排除的端点，请为该端点添加 `x-hidden` 扩展。

    <Info>
      使用 [Swagger Editor](https://editor.swagger.io/) 或 [Mint 命令行工具（mint）](https://www.npmjs.com/package/mint) 验证你的 OpenAPI 文件。
    </Info>
  </Step>

  <Step title="更新导航结构。">
    在 `docs.json` 中，将对 `MDX` 页面的引用替换为 OpenAPI 端点。

    ```json
    "navigation": {
      "groups": [
        {
          "group": "API Reference",
          "openapi": "/path/to/openapi.json",
          "pages": [
            "overview",
            "authentication",
            "introduction",
            "GET /health",
            "quickstart", 
            "POST /users",
            "GET /users/{id}",
            "advanced-features"
          ]
        }
      ]
    }
    ```
  </Step>

  <Step title="移除旧的 MDX 文件。">
    在确认新的导航正常工作后，移除不再需要的 `MDX` 端点文件。
  </Step>
</Steps>

<div id="navigation-patterns">
  ## 导航模式
</div>

您可以自定义 API 文档在导航中的显示方式。

<div id="mixed-content-navigation">
  ### 混合内容导航
</div>

将自动生成的 API 页面与其他页面结合使用：

```json
"navigation": {
  "groups": [
    {
      "group": "API Reference",
      "openapi": "openapi.json",
      "pages": [
        "api/overview",
        "GET /users",
        "POST /users", 
        "api/authentication"
      ]
    }
  ]
}
```

<div id="multiple-api-versions">
  ### 多个 API 版本
</div>

使用选项卡或分组来管理不同的 API 版本：

```json
"navigation": {
  "tabs": [
    {
      "tab": "API v1",
      "openapi": "specs/v1.json"
    },
    {
      "tab": "API v2", 
      "openapi": "specs/v2.json"
    }
  ]
}
```

<div id="when-to-use-individual-mdx-pages">
  ## 何时使用独立的 `MDX` 页面
</div>

在以下情况下建议保留独立的 `MDX` 页面：

- 针对每个端点需要大量自定义内容，例如 React 组件或较长的示例。
- 需要独特的页面布局。
- 为特定端点尝试实验性的文档写作方式。

对于大多数场景，OpenAPI 导航在可维护性和一致性方面表现更佳。
