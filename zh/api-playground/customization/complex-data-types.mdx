当你的 API 接受多种数据格式、包含条件字段或采用继承模式时，OpenAPI 的架构组合关键字可以帮助你为这些灵活的结构编写文档。通过使用 `oneOf`、`anyOf` 和 `allOf`，你可以描述既能处理不同输入类型，又能将多个架构组合为完整数据模型的 API。

<div id="oneof-anyof-allof-keywords">
  ## `oneOf`, `anyOf`, `allOf` 关键字
</div>

对于复杂数据类型，OpenAPI 提供了用于组合架构的关键字：

- `allOf`：组合多个架构（例如合并对象或在基础架构上扩展）。其作用类似于 `and` 运算符。
- `anyOf`：接受与提供的任一架构匹配的数据。其作用类似于 `or` 运算符。
- `oneOf`：接受与提供的架构中恰好一个匹配的数据。其作用类似于 `exclusive-or` 运算符。

<Warning>Mintlify 将 `oneOf` 和 `anyOf` 以相同方式处理，因为它们在实际使用中几乎不会对 API 产生区别。</Warning>

有关这些关键字的详细规范，请参阅 [OpenAPI 文档](https://swagger.io/docs/specification/data-models/oneof-anyof-allof-not/)。

<Info>`not` 关键字目前不受支持。</Info>

<div id="combining-schemas-with-allof">
  ### 使用 `allOf` 组合模式
</div>

当你使用 `allOf` 时，Mintlify 会对你的 OpenAPI 文档进行预处理，以更易读的方式展示复杂组合。比如，当你用 `allOf` 合并两个对象模式时，Mintlify 会将两者的属性合并为一个对象。这在利用 OpenAPI 的可复用[组件](https://swagger.io/docs/specification/components/)时尤其有用。

```yaml
org_with_users:
  allOf:
    - $ref: '#/components/schemas/Org'
    - type: object
      properties:
        users:
          type: array
          description: 包含组织中所有用户的数组
# ...
components:
  schemas:
    Org:
      type: object
      properties:
        id:
          type: string
          description: 组织的 ID
```

<div id="providing-options-with-oneof-and-anyof">
  ### 使用 `oneOf` 和 `anyOf` 提供选项
</div>

当你使用 `oneOf` 或 `anyOf` 时，这些选项会显示在带有选项卡的容器中。请在每个子模式中指定一个 `title` 字段，以为选项命名。例如，下面展示了如何显示两种不同类型的收货地址：

```yaml
delivery_address:
  oneOf:
    - title: StreetAddress
      type: object
      properties:
        address_line_1:
          type: string
          description: 收件人的街道地址
        # ...
    - title: POBox
      type: object
      properties:
        box_number:
          type: string
          description: 邮政信箱编号
        # ...
```
