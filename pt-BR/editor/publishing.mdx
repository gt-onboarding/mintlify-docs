A forma como suas alterações são publicadas depende do branch em que você está trabalhando:

* **Branch de produção**: Publicar atualiza seu site no ar imediatamente.
* **Outros branches**: Publicar cria um pull request para que você revise as alterações antes de irem para produção.

<div id="pull-requests-and-reviewing-changes">
  ## Pull requests e revisão de alterações
</div>

Pull requests permitem que você proponha alterações a partir do seu branch para que outras pessoas as revisem antes de fazer o merge na sua documentação em produção. Isso ajuda a garantir que suas alterações estejam corretas e dá à sua equipe a chance de colaborar no conteúdo.

<Tip>
  Mesmo que você esteja trabalhando sozinho, pull requests são úteis para pré-visualizar alterações e manter um histórico claro de atualizações.
</Tip>

<div id="creating-a-pull-request">
  ### Criando um pull request
</div>

<Steps>
  <Step title="Salve seu trabalho">
    Certifique-se de que todas as suas alterações foram salvas no seu branch usando **Save Changes**.
  </Step>
  <Step title="Crie o pull request">
    Selecione **Publish Pull Request** no canto superior direito do editor.
  </Step>
  <Step title="Adicione um título e uma descrição">
    Escreva um título e uma descrição claros explicando:
    - Quais alterações você fez
    - Por que as fez
    - Quais áreas específicas precisam de revisão
  </Step>
  <Step title="Crie e compartilhe">
    Selecione **Publish Pull Request**. O editor fornecerá um link para visualizar seu pull request.

<div id="reviewing-pull-requests">
  ## Revisando pull requests
</div>

Depois que seu pull request for criado:

1. **Revisar alterações**: Você e os membros da sua equipe podem revisar seu pull request no provedor Git, como GitHub ou GitLab.
2. **Deixar feedback**: Adicione comentários ou solicite alterações.
3. **Fazer alterações adicionais**: Faça alterações adicionais no editor web. Quando você salvar as alterações, o editor as enviará para o seu pull request.
4. **Aprovar**: Aprove o pull request quando estiver satisfeito com as alterações.
5. **Fazer merge**: Faça o merge do pull request quando estiver pronto para implantar suas alterações em produção.
