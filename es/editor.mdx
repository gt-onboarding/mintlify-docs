Accede al editor visual desde tu [panel](https://dashboard.mintlify.com/editor) para gestionar tu documentación directamente en el navegador.

* **Edición WYSIWYG**: Realiza cambios en tu documentación con un editor what-you-see-is-what-you-get (WYSIWYG) que muestra cómo se verá tu contenido al publicarse.
* **Sincronización con Git**: Todos los cambios se sincronizan automáticamente con tu repositorio Git para mantener el control de versiones.
* **Colaboración en equipo**: Varias personas pueden trabajar en la documentación simultáneamente.
* **Integración de componentes**: Agrega avisos, bloques de código y otros componentes con comandos de barra.
* **Sin necesidad de configuración**: Comienza a escribir de inmediato desde tu panel.

<div id="overview">
  ## Descripción general
</div>

Así es como normalmente trabajarás en el editor visual:

<Steps>
  <Step title="Elige tu rama">
    Crea una rama o realiza cambios directamente en tu rama de despliegue. Recomendamos crear una rama para poder previsualizar tus cambios antes de que se publiquen.
  </Step>

  <Step title="Abre tu archivo">
    Ve a un archivo existente en la barra lateral o crea uno nuevo con el explorador de archivos.
  </Step>

  <Step title="Edita tu contenido">
    Haz cambios en el editor visual. Pulsa <kbd>/</kbd> para abrir el menú de componentes.
  </Step>

  <Step title="Previsualiza tus cambios">
    El modo visual te muestra cómo se verán tus cambios en tu sitio en producción. Úsalo para verificar que todo esté correcto.
  </Step>

  <Step title="Publica tus cambios">
    Si trabajas en tu rama de despliegue, publica tus cambios directamente desde el editor visual. En otras ramas, crea un pull request para revisión antes de publicar.
  </Step>
</Steps>

</div>

<div id="organize-your-navigation">
  ### Organiza la navegación
</div>

Edita tu archivo `docs.json` para agregar páginas nuevas y quitar las eliminadas de la navegación de tu sitio. Consulta [Navegación](/es/organize/navigation) para obtener más información sobre cómo organizar páginas.

**Ejemplo: agregar una página Themes al grupo Profile**

En este ejemplo, creaste una página nueva titulada Themes y quieres agregarla al grupo Profile de tu documentación. Agrega la ruta de la nueva página al arreglo `pages` del grupo `Profile` en tu archivo `docs.json` para que aparezca en la navegación de tu sitio.

```json title="Adding a Themes page to the Profile group" highlight="18"
{
    "navigation": {
        "groups": [
            {
                "group": "Comenzar",
                "pages": [
                    "index",
                    "quickstart",
                    "installation"
                ]
            },
            {
                "group": "Perfil",
                "pages": [
                    "settings",
                    "account-types",
                    "dashboard",
                    "themes"
                ]
            }
        ]
    }
}
```

<div id="edit-content">
  ### Editar contenido
</div>

Realiza cambios en tus páginas usando el modo visual o el modo Markdown en el editor.

En el modo visual, presiona <kbd>/</kbd> para abrir el menú de componentes. Agrega bloques de contenido, avisos, bloques de código y otros componentes para personalizar tu documentación.

<Frame>
  <img src="/images/editor/component-menu-light.png" alt="El menú de componentes desplegado resaltado en el editor visual de Mintlify" className="block dark:hidden" />

  <img src="/images/editor/component-menu-dark.png" alt="El menú de componentes desplegado resaltado en el editor visual de Mintlify" className="hidden dark:block" />
</Frame>

En el modo Markdown, editas directamente el `MDX` de tus páginas. Esto puede ser útil cuando necesitas:

* Definir propiedades específicas de los componentes
* Trabajar con componentes anidados complejos
* Copiar y pegar contenido `MDX` de otras fuentes

Consulta [Formato de texto](/es/create/text) y [Formato de código](/es/create/code) para obtener más información sobre cómo escribir con sintaxis Markdown.

  ## Publica tus cambios
</div>

La rama en la que trabajes determina cómo el editor publica tus cambios:

* **Rama de deployment**: Al publicar, tu sitio en producción se actualiza de inmediato.
* **Otras ramas**: Al publicar, se crea una pull request para que puedas revisar los cambios antes de desplegarlos a producción.

<div id="pull-requests-and-reviewing-changes">
  ### Pull requests y revisión de cambios
</div>

Los pull requests te permiten proponer cambios desde tu rama para que otras personas los revisen antes de fusionarlos en tu documentación publicada. Esto ayuda a garantizar que tus cambios sean correctos y le da a tu equipo la oportunidad de colaborar en el contenido.

<Tip>
  Incluso si trabajas solo, los pull requests son útiles para previsualizar cambios y mantener un historial claro de actualizaciones.
</Tip>

<div id="create-a-pull-request">
  #### Crear una pull request
</div>

<Steps>
  <Step title="Guarda tu trabajo">
    Selecciona **Save Changes** para guardar todos los cambios en tu rama.
  </Step>

  <Step title="Crea la pull request">
    Selecciona **Publish Pull Request** desde la barra de herramientas del editor.
  </Step>

  <Step title="Agrega un título y una descripción">
    Escribe un título y una descripción claros que expliquen:

    * Qué cambios realizaste
    * Por qué los hiciste
    * Cualquier área específica que necesite revisión
  </Step>

  <Step title="Crea y comparte">
    Selecciona **Publish Pull Request**. El editor proporcionará un enlace para ver tu pull request.

    <Frame>
      <img src="/images/editor/pull-request-light.png" alt="Botón Publish Pull Request resaltado en el editor visual de Mintlify" className="block dark:hidden" />

      <img src="/images/editor/pull-request-dark.png" alt="Botón Publish Pull Request resaltado en el editor visual de Mintlify" className="hidden dark:block" />
    </Frame>
  </Step>
</Steps>

<div id="review-pull-requests">
  #### Revisar pull requests
</div>

Una vez creado tu pull request:

1. **Revisar cambios**: Tú y los miembros de tu equipo pueden revisar tu pull request en tu proveedor de Git como GitHub o GitLab.
2. **Dejar comentarios**: Agrega comentarios o solicita cambios.
3. **Realizar cambios adicionales**: Realiza cambios adicionales en el editor visual. Cuando guardes los cambios, el editor los enviará a tu pull request.
4. **Aprobar**: Aprueba el pull request cuando estés satisfecho con los cambios.
5. **Fusionar**: Fusiona el pull request cuando estés listo para desplegar tus cambios a producción.

<div id="troubleshooting">
  ## Solución de problemas
</div>

Encuentra soluciones a problemas comunes que podrías encontrar al usar el editor visual.

<AccordionGroup>
  <Accordion title="Los cambios no aparecen después de publicar">
    **Posibles causas:**

    * El despliegue aún está en proceso
    * Problemas de caché del navegador
    * Errores de compilación o despliegue

    **Soluciones:**

    1. Verifica el estado del despliegue en tu panel.
    2. Fuerza la recarga del navegador (<kbd>Ctrl</kbd> + <kbd>F5</kbd> o <kbd>Cmd</kbd> + <kbd>Shift</kbd> + <kbd>R</kbd>)
    3. Borra la caché de tu navegador.
  </Accordion>

  <Accordion title="Errores de permisos al publicar">
    **Posibles causas:**

    * Permisos insuficientes en el repositorio de Git
    * Problemas de autenticación con tu proveedor de Git

    **Soluciones:**

    1. Verifica que tengas el acceso correcto al repositorio.
    2. Comprueba que tu integración con Git esté configurada correctamente.
    3. Revisa la documentación de [Permisos del editor](/es/dashboard/permissions).
  </Accordion>

  <Accordion title="Problemas al cargar el editor">
    **Posibles causas:**

    * Problemas de conectividad de red
    * Repositorios de documentación muy grandes

    **Soluciones:**

    1. Verifica tu conexión a internet.
    2. Recarga la página.
    3. Contacta con soporte si el problema persiste.
  </Accordion>

  <Accordion title="Los archivos no se cargan o muestran errores">
    **Posibles causas:**

    * Sintaxis MDX no válida en los archivos
    * Archivos faltantes o dañados
    * Archivos de gran tamaño que provocan tiempos de espera

    **Soluciones:**

    1. Revisa la sintaxis del archivo para detectar errores de formato en MDX.
    2. Verifica que el archivo exista en tu repositorio.
  </Accordion>
</AccordionGroup>
