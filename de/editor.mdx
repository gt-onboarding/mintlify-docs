<div id="introduction">
  ## Einführung
</div>

Der Web-Editor ist eine visuelle Oberfläche zum Erstellen, Bearbeiten und Überprüfen von Dokumentation direkt in deinem Browser.

- **Visuelle Bearbeitung**: Nimm Änderungen an deiner Dokumentation mit einem What-you-see-is-what-you-get-(WYSIWYG)-Editor vor, der zeigt, wie dein Inhalt bei der Veröffentlichung aussieht.
- **Git-Synchronisierung**: Alle Änderungen werden automatisch mit deinem Git-Repository synchronisiert, um die Versionskontrolle sicherzustellen.
- **Zusammenarbeit in Echtzeit**: Mehrere Teammitglieder können gleichzeitig an der Dokumentation arbeiten, und Änderungen sind für alle sichtbar.
- **Keine Einrichtung erforderlich**: Starte direkt über dein Dashboard mit dem Schreiben.

<div id="web-editor-flow">
  ### Web-Editor-Ablauf
</div>

So arbeiten Sie typischerweise im Web-Editor:

<Steps>
  <Step title="Wählen Sie Ihren Branch">
    Erstellen Sie einen Branch oder nehmen Sie Änderungen direkt an Ihrem Deployment-Branch vor.
  </Step>

  <Step title="Öffnen Sie Ihre Datei">
    Navigieren Sie zu einer vorhandenen Datei oder erstellen Sie eine neue.
  </Step>

  <Step title="Bearbeiten Sie Ihren Inhalt">
    Nehmen Sie Änderungen im Web-Editor entweder im visuellen Modus oder im Markdown-Modus vor.
  </Step>

  <Step title="Änderungen prüfen">
    Sehen Sie im visuellen Modus, wie Ihre Änderungen angezeigt werden.
  </Step>

  <Step title="Veröffentlichen Sie Ihre Änderungen">
    Wenn Sie auf Ihrem Deployment-Branch arbeiten, veröffentlichen Sie Ihre Änderungen direkt aus dem Web-Editor. Auf anderen Branches veröffentlichen Sie Ihre Änderungen über eine Pull-Request.
  </Step>
</Steps>

<div id="visual-mode">
  ### Visueller Modus
</div>

Der visuelle Modus bietet ein WYSIWYG-Erlebnis, bei dem die im Editor vorgenommenen Änderungen genau so auf Ihrer Dokumentationsseite veröffentlicht werden. Dieser Modus ist ideal, wenn Sie in Echtzeit sehen möchten, wie Ihre Änderungen aussehen.

<div id="markdown-mode">
  ### Markdown-Modus
</div>

Der Markdown-Modus bietet direkten Zugriff auf den zugrunde liegenden `MDX`-Code Ihrer Dokumentation. `MDX` kombiniert die Markdown-Syntax mit React-Komponenten und gibt Ihnen die volle Kontrolle über die Struktur Ihrer Inhalte. Dieser Modus ist ideal, wenn Sie präzise Kontrolle über Komponenten­eigenschaften benötigen oder wenn Sie es vorziehen, in Markdown-Syntax zu schreiben.

<div id="git-fundamentals">
  ## Git-Grundlagen
</div>

Der Web-Editor führt im Hintergrund Git-Operationen aus. Das Verständnis dieser Konzepte hilft Ihnen, effektiver mit dem Web-Editor zu arbeiten und mit Teammitgliedern zusammenzuarbeiten, die lokal arbeiten.

<AccordionGroup>
  <Accordion title="Repository">
    Die Quelle Ihrer Dokumentation, in der alle Dateien und deren Historie gespeichert sind. Der Web-Editor verbindet sich mit Ihrem Repository, um auf Inhalte zuzugreifen und sie zu bearbeiten.
  </Accordion>

  <Accordion title="Commit">
    Eine gespeicherte Momentaufnahme Ihrer Änderungen zu einem bestimmten Zeitpunkt.
  </Accordion>

  <Accordion title="Branch">
    Ein separater Arbeitsbereich, in dem Sie Änderungen vornehmen können, ohne Ihre Live-Dokumentation zu beeinflussen. Denken Sie daran als sichere Sandbox für Experimente und größere Updates.
  </Accordion>

  <Accordion title="Deployment branch">
    Der Haupt-Branch, der Ihre Live-Dokumentationsinhalte enthält. Änderungen an diesem Branch werden automatisch auf Ihrer Dokumentationsseite veröffentlicht. Oft als `main` bezeichnet.
  </Accordion>

  <Accordion title="Pull request">
    Eine Möglichkeit, das Zusammenführen Ihrer Branch-Änderungen in die Live-Dokumentation vorzuschlagen. Ermöglicht Überprüfung und Diskussion, bevor Änderungen live gehen. Häufig als PR bezeichnet.
  </Accordion>

  <Accordion title="Diff">
    Ein Diff (oder Unterschied) zeigt die konkreten Änderungen zwischen zwei Versionen einer Datei. Beim Überprüfen von Pull Requests heben Diffs hervor, was hinzugefügt, entfernt oder geändert wurde.
  </Accordion>
</AccordionGroup>

<div id="what-the-web-editor-automates">
  ### Was der Web-Editor automatisiert
</div>

Der Web-Editor verbindet sich über unsere [GitHub-App](/de/settings/github) oder [GitLab-Integration](/de/settings/gitlab) mit Ihrem Git-Repository und übernimmt Git-Aufgaben automatisch. Wenn Sie:

- **Eine Datei öffnen**: wird die neueste Version aus Ihrem Repository abgerufen.
- **Änderungen in einer Datei vornehmen**: werden Ihre Änderungen als Entwurf nachverfolgt, der zu einem Commit werden kann.
- **Änderungen speichern**: wird ein Commit mit Ihren Änderungen erstellt.
- **Einen Branch erstellen**: wird ein neuer Branch in Ihrem Repository erstellt.
- **Auf Ihrem Deployment-Branch veröffentlichen**: wird ein Commit erstellt und direkt auf Ihren Deployment-Branch gepusht.
- **Auf anderen Branches veröffentlichen**: wird ein Commit erstellt und eine Pull-Request eröffnet.

<div id="when-to-use-branches">
  ### Wann Branches verwendet werden sollten
</div>

Branches ermöglichen es Ihnen, an Änderungen zu arbeiten, ohne den Inhalt Ihrer Live‑Site zu beeinflussen. Wenn Ihre Änderungen bereit sind, können Sie sie per Pull Request in Ihren Deployment‑Branch mergen.

**Best Practice: Arbeiten Sie immer mit Branches.** So bleibt Ihre Live‑Dokumentation stabil, und saubere Review‑Workflows sind gewährleistet.

<div id="creating-a-branch">
  ## Einen Branch erstellen
</div>

1. Wählen Sie den Branchnamen in der Editor‑Toolbar aus (standardmäßig meist `main`).
2. Wählen Sie **New Branch**.
3. Geben Sie einen aussagekräftigen Namen für Ihren Branch ein, z. B. `update-getting-started` oder `fix-installation-steps`.
4. Wählen Sie **Create Branch**.

<Note>
  Möglicherweise müssen Sie Ihren neuen Branch im Dropdown-Menü auswählen, wenn der Editor nicht automatisch zu ihm wechselt.
</Note>

<div id="saving-changes-on-a-branch">
  ### Änderungen in einem Branch speichern
</div>

Um Ihre Änderungen in einem Branch zu speichern, klicken Sie oben rechts im Editor auf **Save Changes**. Dadurch wird ein Commit mit Ihren Änderungen erstellt und in Ihren Branch gepusht.

<div id="switching-branches">
  ### Branches wechseln
</div>

1. Wählen Sie den Namen des aktuellen Branches in der Editor-Toolbar aus.
2. Wählen Sie den Branch, zu dem Sie wechseln möchten, aus dem Dropdown-Menü.

<Tip>
  Alle nicht gespeicherten Änderungen gehen beim Wechseln des Branches verloren. Speichern oder veröffentlichen Sie Ihre Arbeit, bevor Sie wechseln.
</Tip>

</div>

<div id="updating-navigation">
  ### Navigation aktualisieren
</div>

Fügen Sie Seiten hinzu, entfernen Sie sie und ordnen Sie sie in Ihrer Navigation neu, indem Sie die Datei `docs.json` bearbeiten:

1. Öffnen Sie im Dateiexplorer Ihre `docs.json`.
2. Aktualisieren Sie die Eigenschaft `navigation`, um die gewünschte Navigationsstruktur zu erhalten. Weitere Informationen finden Sie unter [Navigation](/de/navigation).

Dieses Beispiel zeigt, wie eine Seite „Themes“ zur Gruppe „Profile“ hinzugefügt wird.

```json title="Adding a Themes page" highlight="18"
{
    "navigation": {
        "groups": [
            {
                "group": "Getting started",
                "pages": [
                    "index",
                    "quickstart",
                    "installation"
                ]
            },
            {
                "group": "Profile",
                "pages": [
                    "settings",
                    "account-types",
                    "dashboard",
                    "themes"
                ]
            }
        ]
    }
}
```

<div id="publishing-changes">
  ## Änderungen veröffentlichen
</div>

Wähle die Schaltfläche **Publish**, um deine Änderungen zu speichern und zu veröffentlichen.

Wie deine Änderungen veröffentlicht werden, hängt davon ab, an welchem Branch du arbeitest:

- **Deployment-Branch**: Aktualisiert deine Live‑Site sofort.
- **Andere Branches**: Erstellt eine Pull-Request, damit du die Änderungen prüfen kannst, bevor sie live gehen.

<div id="pull-requests-and-reviewing-changes">
  ## Pull Requests und Änderungen prüfen
</div>

Pull Requests ermöglichen es dir, Änderungen aus deinem Branch vorzuschlagen, damit andere sie begutachten können, bevor sie in deine Live-Dokumentation gemergt werden. So stellst du die Korrektheit deiner Änderungen sicher und gibst deinem Team die Möglichkeit, gemeinsam an Inhalten zu arbeiten.

<Tip>
  Auch wenn du allein arbeitest, sind Pull Requests hilfreich, um Änderungen vor der Veröffentlichung in der Vorschau zu sehen und eine klare Historie der Aktualisierungen zu behalten.
</Tip>

<div id="creating-a-pull-request-on-a-branch">
  ### Einen Pull Request auf einem Branch erstellen
</div>

Wenn Sie bereit sind, Änderungen aus Ihrem Branch zu veröffentlichen:

1. Stellen Sie sicher, dass alle Änderungen in Ihrem Branch gespeichert sind.
2. Wählen Sie **Publish Pull Request** oben rechts im Editor.
3. Fügen Sie einen Titel und eine Beschreibung für Ihren Pull Request hinzu. Ein prägnanter Titel und eine klare Beschreibung helfen den Reviewern, Ihre Änderungen nachzuvollziehen.
4. Wählen Sie **Publish Pull Request**.

<Frame>
  <img src="/images/editor/pull-request-light.png" alt="Hervorgehobene Schaltfläche „Publish Pull Request“ im Mintlify Web-Editor" className="block dark:hidden" />

  <img src="/images/editor/pull-request-dark.png" alt="Hervorgehobene Schaltfläche „Publish Pull Request“ im Mintlify Web-Editor" className="hidden dark:block" />
</Frame>

Der Web-Editor erstellt in Ihrem Git-Repository einen Pull Request und stellt einen Link zur Ansicht bereit.

<div id="reviewing-pull-requests">
  ### Pull Requests überprüfen
</div>

Sobald Ihr Pull Request erstellt wurde:

1. **Änderungen prüfen**: Sie und Ihre Teammitglieder können Ihren Pull Request in Ihrem Git-Anbieter wie GitHub oder GitLab prüfen.
2. **Weitere Änderungen vornehmen**: Nehmen Sie nach der Überprüfung alle Änderungen in Ihrem Web-Editor vor. Das Speichern Ihrer Änderungen aktualisiert den Pull Request.
3. **Zusammenführen, wenn bereit**: Wenn Ihr Pull Request bereit ist, führen Sie ihn zusammen, um die Änderungen auf Ihrer Live-Dokumentationsseite zu veröffentlichen.

<div id="troubleshooting">
  ## Fehlerbehebung
</div>

Hier finden Sie Lösungen für häufige Probleme, die im Web-Editor auftreten können.

<AccordionGroup>
  <Accordion title="Änderungen werden nach der Veröffentlichung nicht angezeigt">
    **Mögliche Ursachen:**

    - Bereitstellung läuft noch
    - Cache-Probleme in Ihrem Browser

    **Lösungen:**

    1. Prüfen Sie den Bereitstellungsstatus in Ihrem Mintlify-Dashboard.
    2. Führen Sie einen Hard-Refresh Ihres Browsers durch (<kbd>Ctrl</kbd> + <kbd>F5</kbd> oder <kbd>Cmd</kbd> + <kbd>Shift</kbd> + <kbd>R</kbd>).
    3. Leeren Sie den Browser-Cache.
  </Accordion>

  <Accordion title="Berechtigungsfehler beim Veröffentlichen">
    **Mögliche Ursachen:**

    - Unzureichende Berechtigungen für das Git-Repository
    - Authentifizierungsprobleme mit Ihrem Git-Anbieter

    **Lösungen:**

    1. Stellen Sie sicher, dass Sie die richtigen Zugriffsrechte auf das Repository haben.
    2. Überprüfen Sie, ob Ihre Git-Integration korrekt konfiguriert ist.
    3. Lesen Sie die Dokumentation zu [Editorberechtigungen](/de/advanced/dashboard/permissions).
  </Accordion>

  <Accordion title="Editor lädt nicht ordnungsgemäß">
    **Mögliche Ursachen:**

    - Netzwerkverbindungsprobleme
    - Große Dokumentations-Repositories

    **Lösungen:**

    1. Überprüfen Sie Ihre Internetverbindung.
    2. Laden Sie die Seite neu und versuchen Sie es erneut.
    3. Kontaktieren Sie den Support, wenn das Problem weiterhin besteht.
  </Accordion>
</AccordionGroup>
