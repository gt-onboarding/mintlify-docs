Wenn Sie derzeit einzelne `MDX`-Seiten für Ihre API-Endpunkte verwenden, können Sie auf automatisch generierte Seiten aus Ihrer OpenAPI-Spezifikation umstellen und dabei die Anpassbarkeit einzelner Seiten beibehalten. Das hilft, die Anzahl der zu pflegenden Dateien zu reduzieren und die Konsistenz Ihrer API-Dokumentation zu verbessern.

Sie können Metadaten und Inhalte für jeden Endpunkt in Ihrer OpenAPI-Spezifikation definieren und die Endpunkte in Ihrer Navigation nach Bedarf platzieren.

<div id="cli-migration">
  ## CLI-Migration
</div>

Der Befehl `mint migrate-mdx` ist die empfohlene Methode, um von MDX-Endpunktseiten auf automatisch generierte Seiten zu migrieren.

Dieser Befehl:

- Analysiert Ihre `docs.json`-Navigationsstruktur.
- Identifiziert MDX-Seiten, die OpenAPI-Endpunktseiten erzeugen.
- Extrahiert Inhalte aus MDX-Dateien und verschiebt sie in die `x-mint`-Erweiterung Ihrer OpenAPI-Spezifikation.
- Aktualisiert Ihre `docs.json`, sodass die OpenAPI-Endpunkte direkt statt der MDX-Dateien referenziert werden.
- Löscht die ursprünglichen MDX-Endpunktdateien.

<Info>
  Wenn Sie bereits `x-mint` für einen Endpunkt definiert haben und zusätzlich eine MDX-Seite mit Inhalt für diesen Endpunkt existiert, überschreibt der MDX-Inhalt die bestehenden `x-mint`-Einstellungen.

  Wenn Sie mehrere MDX-Seiten für denselben Endpunkt mit unterschiedlichen Inhalten haben, verwendet das Skript den Inhalt der Seite, die in Ihrer `docs.json` zuletzt aufgeführt ist.

  Das Migrationswerkzeug bietet keine Vorschau der Änderungen, bevor sie angewendet werden.
</Info>

<Steps>
  <Step title="Bereiten Sie Ihre OpenAPI-Spezifikation vor.">
    Stellen Sie sicher, dass Ihre OpenAPI-Spezifikation gültig ist und alle Endpunkte enthält, die Sie dokumentieren möchten.

    Alle MDX-Seiten, die Sie migrieren möchten, müssen im Frontmatter `openapi:` einen Verweis auf einen Endpunkt enthalten.

    <Tip>
      Validieren Sie Ihre OpenAPI-Datei mit dem [Swagger Editor](https://editor.swagger.io/) oder der [Mint-CLI](https://www.npmjs.com/package/mint).
    </Tip>
  </Step>

  <Step title="Installieren Sie die Mint-CLI">
    Installieren oder aktualisieren Sie bei Bedarf die [Mint-CLI](/de/installation).
  </Step>

  <Step title="Führen Sie den Migrationsbefehl aus.">
    ```bash
    mint migrate-mdx
    ```
  </Step>
</Steps>

<div id="manual-migration-steps">
  ## Manuelle Migrationsschritte
</div>

<Steps>
  <Step title="Bereiten Sie Ihre OpenAPI-Spezifikation vor.">
    Stellen Sie sicher, dass Ihre OpenAPI-Spezifikation gültig ist und alle Endpunkte enthält, die Sie dokumentieren möchten.

    Für Endpunkte, deren Metadaten oder Inhalte Sie anpassen möchten, fügen Sie dem Endpunkt die Erweiterung `x-mint` hinzu. Siehe [x-mint-Erweiterung](/de/api-playground/openapi-setup#x-mint-extension) für weitere Details.

    Für Endpunkte, die Sie aus Ihrer Dokumentation ausschließen möchten, fügen Sie dem Endpunkt die Erweiterung `x-hidden` hinzu.

    <Info>
      Validieren Sie Ihre OpenAPI-Datei mit dem [Swagger Editor](https://editor.swagger.io/) oder der [Mint-CLI](https://www.npmjs.com/package/mint).
    </Info>
  </Step>

  <Step title="Aktualisieren Sie Ihre Navigationsstruktur.">
    Ersetzen Sie `MDX`-Seitenverweise in Ihrer `docs.json` durch OpenAPI-Endpunkte.

    ```json
    "navigation": {
      "groups": [
        {
          "group": "API Reference",
          "openapi": "/path/to/openapi.json",
          "pages": [
            "overview",
            "authentication",
            "introduction",
            "GET /health",
            "quickstart", 
            "POST /users",
            "GET /users/{id}",
            "advanced-features"
          ]
        }
      ]
    }
    ```
  </Step>

  <Step title="Entfernen Sie alte MDX-Dateien.">
    Nachdem Sie überprüft haben, dass Ihre neue Navigation korrekt funktioniert, entfernen Sie die `MDX`-Endpunktdateien, die Sie nicht mehr benötigen.
  </Step>
</Steps>

<div id="navigation-patterns">
  ## Navigationsmuster
</div>

Sie können steuern, wie Ihre API-Dokumentation in der Navigation erscheint.

<div id="mixed-content-navigation">
  ### Navigation mit gemischten Inhalten
</div>

Kombinieren Sie automatisch generierte API-Seiten mit anderen Seiten:

```json
"navigation": {
  "groups": [
    {
      "group": "API-Referenz",
      "openapi": "openapi.json",
      "pages": [
        "api/overview",
        "GET /users",
        "POST /users", 
        "api/authentication"
      ]
    }
  ]
}
```

<div id="multiple-api-versions">
  ### Mehrere API-Versionen
</div>

Organisieren Sie unterschiedliche API-Versionen mithilfe von Tabs oder Gruppen:

```json
"navigation": {
  "tabs": [
    {
      "tab": "API v1",
      "openapi": "specs/v1.json"
    },
    {
      "tab": "API v2", 
      "openapi": "specs/v2.json"
    }
  ]
}
```

<div id="when-to-use-individual-mdx-pages">
  ## Wann einzelne `MDX`-Seiten verwenden
</div>

Behalten Sie einzelne `MDX`-Seiten bei, wenn Sie Folgendes benötigen:

- Umfangreiche, maßgeschneiderte Inhalte pro Endpoint, etwa React-Komponenten oder ausführliche Beispiele.
- Individuelle Seitenlayouts.
- Experimentelle Dokumentationsansätze für bestimmte Endpoints.

Für die meisten Anwendungsfälle bietet die OpenAPI-Navigation bessere Wartbarkeit und Konsistenz.
