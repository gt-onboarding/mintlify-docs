<div id="overview">
  ## Übersicht
</div>

Die API-Spielwiese ist eine interaktive Umgebung, in der Benutzer Ihre API-Endpunkte testen und erkunden können. Entwickler können API-Anfragen erstellen, senden und Antworten ansehen, ohne Ihre Dokumentation zu verlassen.

<Frame>
  <img src="/images/playground/API-playground-light.png" alt="API-Spielwiese für den Endpunkt zum Auslösen eines Updates." className="block dark:hidden" />

  <img src="/images/playground/API-playground-dark.png" alt="API-Spielwiese für den Endpunkt zum Auslösen eines Updates." className="hidden dark:block" />
</Frame>

Die Spielwiese wird automatisch aus Ihrer OpenAPI-Spezifikation oder Ihrem AsyncAPI-Schema generiert, sodass alle Aktualisierungen Ihrer API automatisch in der Spielwiese reflektiert werden. Sie können auch manuell API-Referenzseiten erstellen, nachdem Sie in Ihrer `docs.json` eine Basis-URL und eine Authentifizierungsmethode definiert haben.

Wir empfehlen, Ihre API-Spielwiese auf Basis einer OpenAPI-Spezifikation zu erzeugen. Weitere Informationen zur Erstellung Ihres OpenAPI-Dokuments finden Sie unter [OpenAPI Setup](/de/api-playground/openapi-setup).

<div id="getting-started">
  ## Erste Schritte
</div>

<Steps>
  <Step title="Fügen Sie Ihre OpenAPI-Spezifikationsdatei hinzu.">
    <Info>
      Stellen Sie sicher, dass Ihre OpenAPI-Spezifikationsdatei gültig ist – beispielsweise mit dem [Swagger Editor](https://editor.swagger.io/) oder der [Mint CLI](https://www.npmjs.com/package/mint).
    </Info>

    ```bash {3}
    /your-project
      |- docs.json
      |- openapi.json
    ```
  </Step>

  <Step title="Konfigurieren Sie `docs.json`.">
    Aktualisieren Sie Ihre `docs.json`, damit sie auf Ihre OpenAPI-Spezifikation verweist. Fügen Sie einem Navigationselement die Eigenschaft `openapi` hinzu, um Ihre Dokumentation automatisch mit Seiten für jeden im OpenAPI-Dokument definierten Endpunkt zu befüllen.

    Dieses Beispiel erzeugt eine Seite für jeden in `openapi.json` definierten Endpunkt und ordnet sie in Ihrer Navigation der Gruppe „API reference“ zu.

    ```json
    "navigation": {
      "groups": [
        {
          "group": "API reference",
          "openapi": "openapi.json"
        }
      ]
    }
    ```

    Um nur für bestimmte Endpunkte Seiten zu generieren, listen Sie die Endpunkte in der Eigenschaft `pages` des Navigationselements auf.

    Dieses Beispiel generiert Seiten ausschließlich für die Endpunkte `GET /users` und `POST /users`. Um weitere Endpunktseiten zu erzeugen, fügen Sie zusätzliche Endpunkte zum `pages`-Array hinzu.

    ```json
    "navigation": {
      "groups": [
          {
            "group": "API reference",
            "openapi": "openapi.json",
            "pages": [
              "GET /users",
              "POST /users"
            ]
          }
      ]
    }
    ```
  </Step>
</Steps>

<div id="customizing-your-playground">
  ## Anpassen Ihrer Spielwiese
</div>

Sie können Ihre API-Spielwiese anpassen, indem Sie die folgenden Eigenschaften in Ihrer `docs.json` definieren.

<ResponseField name="playground" type="object">
  Konfigurationen für die API-Spielwiese.

  <Expandable title="playground" defaultOpen="True">
    <ResponseField name="display" type="&#x22;interactive&#x22; | &#x22;simple&#x22; | &#x22;none&#x22;">
      Der Anzeigemodus der API-Spielwiese.

      * `"interactive"`: Die interaktive Spielwiese anzeigen.
      * `"simple"`: Einen kopierbaren Endpunkt ohne Spielwiese anzeigen.
      * `"none"`: Nichts anzeigen.

      Standardmäßig `interactive`.
    </ResponseField>

    <ResponseField name="proxy" type="boolean" defaultOpen="True">
      Ob API-Anfragen über einen Proxy-Server geleitet werden. Standardmäßig `true`.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="examples" type="object">
  Konfigurationen für die automatisch generierten API-Beispiele.

  <Expandable title="examples" defaultOpen="True">
    <ResponseField name="languages" type="array of string">
      Sprachen für die automatisch generierten API-Snippets.

      Die Sprachen werden in der angegebenen Reihenfolge angezeigt.
    </ResponseField>

    <ResponseField name="defaults" type="&#x22;required&#x22; | &#x22;all&#x22;">
      Ob optionale Parameter in API-Beispielen angezeigt werden. Standardmäßig `all`.
    </ResponseField>

    <ResponseField name="prefill" type="boolean">
      Ob die API-Spielwiese mit Daten aus Schema-Beispielen vorbefüllt wird. Wenn aktiviert, füllt die Spielwiese Anforderungsfelder automatisch mit Beispielwerten aus Ihrer OpenAPI-Spezifikation. Standardmäßig `false`.
    </ResponseField>
  </Expandable>
</ResponseField>

<div id="example-configuration">
  ### Beispielkonfiguration
</div>

```json
{
 "api": {
   "playground": {
     "display": "interactive"
   },
   "examples": {
     "languages": ["curl", "python", "javascript"],
     "defaults": "required",
     "prefill": true
   }
 }
}
```

Dieses Beispiel konfiguriert die API-Spielwiese so, dass sie interaktiv ist und Beispielcode für cURL, Python und JavaScript enthält. In den Codeausschnitten werden nur die erforderlichen Parameter angezeigt, und die Spielwiese füllt den Request-Body mit Beispielwerten vor.

<div id="custom-endpoint-pages">
  ### Benutzerdefinierte Endpunktseiten
</div>

Wenn Sie mehr Kontrolle über Ihre API-Dokumentation benötigen, verwenden Sie die `x-mint`-Erweiterung in Ihrer OpenAPI-Spezifikation oder erstellen Sie einzelne `MDX`-Seiten für Ihre Endpunkte.

Beide Optionen ermöglichen Ihnen:

* Seitenmetadaten anpassen
* Zusätzliche Inhalte wie Beispiele hinzufügen
* Das Verhalten der API-Spielwiese pro Seite steuern

Die `x-mint`-Erweiterung wird empfohlen, damit Ihre gesamte API-Dokumentation automatisch aus Ihrer OpenAPI-Spezifikation generiert und in einer Datei verwaltet wird.

Einzelne `MDX`-Seiten werden für kleine APIs oder dann empfohlen, wenn Sie Änderungen seitenweise ausprobieren möchten.

Weitere Informationen finden Sie unter [x-mint-Erweiterung](/de/api-playground/openapi-setup#x-mint-extension) und [MDX-Einrichtung](/de/api-playground/mdx/configuration).
